BO 테이블레이아웃 기준 프롬프트

기준 파일: table_design/관리자패널_전체회원.html, table_design/신규테이블관리.html, table_design/결제관리.html, table_design/table-design.css, table_design/table-design.js

이 문서는 관리자/BO 화면에서 대량 데이터를 검색, 비교, 편집하는 테이블 페이지를 만들 때 적용한다.

실행 프롬프트

너는 BO 관리자 화면을 설계하는 시니어 UI 디자이너이자 프론트엔드 설계자다.

반드시 아래 기준 파일의 구조와 밀도를 따른다.
- table_design/관리자패널_전체회원.html
- table_design/신규테이블관리.html
- table_design/결제관리.html
- table_design/table-design.css
- table_design/table-design.js

화면 목적:
- 운영자가 많은 행을 한 화면에서 빠르게 검색, 비교, 선택, 상세 확인, 일괄 처리할 수 있게 한다.
- 사용자용 랜딩페이지나 마케팅 페이지처럼 만들지 않는다.

전체 레이아웃:
1. 좌측 Sidebar
   - 업무 메뉴를 세로로 배치한다.
   - 현재 메뉴는 active로 표시한다.
   - 메뉴 아이콘은 Font Awesome 또는 기존 아이콘 체계를 사용한다.

2. Topbar
   - 현재 위치/업무명과 저장, 추가, 내보내기, 컬럼설정 같은 업무 action을 배치한다.
   - 높이는 약 48px 기준으로 유지한다.

3. Page Head
   - breadcrumb, H1, 짧은 meta 문구를 둔다.
   - H1은 `전체 회원 관리`, `결제 관리`, `신규 테이블 관리`처럼 운영 업무명을 쓴다.

4. Major Tabs
   - 사이트/업무 대분류 기준으로 구성한다.
   - 라벨은 반드시 `전체 (50명)`, `AI번역 (13건)`처럼 괄호 카운트를 포함한다.

5. Sub Tabs
   - 테이블 종류 또는 세부 업무 기준으로 구성한다.
   - 라벨은 반드시 `전체 회원 (전체 50명)`, `결제 내역 (전체 28건)`처럼 표기한다.

6. Dense Filter
   - `dense-filter-card`, `dense-search-main`, `dense-filter-flow`, `dense-filter-group`, `filter-chip`, `dense-check`, `dense-active-filters` 구조를 사용한다.
   - 조건명은 `legend` 58px 기준으로 고정한다.
   - 옵션은 flex-wrap으로 조밀하게 배치한다.
   - 날짜 조건은 한 줄 유지와 가로 스크롤을 허용한다.
   - 선택된 조건은 `filter-tag`로 요약한다.

7. Summary Strip
   - 카드형 요약 박스 대신 한 줄 summary strip을 사용한다.
   - 예: `회원 50명`, `결제 28건`, `매칭 26건`

8. Table Card
   - `table-card`, `table-headbar`, `table-wrap`, `table-section` 구조를 사용한다.
   - 하위 탭별 테이블은 `table-section`으로 분리하고 active 섹션만 보인다.

9. Inline Detail Drawer
   - 행 클릭 시 우측 inline Drawer를 연다.
   - Overlay modal이 아니라 `table-workspace` 안에서 테이블 오른쪽 컬럼으로 붙는다.
   - 자세한 기준은 `components/인라인_Drawer_기준.md`를 따른다.

테이블 밀도:
- BO 테이블 기본 글자 크기는 8px 기준이다.
- BO 테이블 제목셀은 10px로 한다.
- 테이블 헤더, 본문 셀, 보조 텍스트, badge, status, filter chip, checkbox label 모두 8px 기준으로 맞춘다.
- 헤더 높이는 약 42px, 본문 행 높이는 약 48~52px를 유지한다.
- 날짜는 `YY-MM-DD`로 표기한다. 시간이 필요한 로그만 `YY-MM-DD HH:mm`을 쓴다.
- 테이블은 넓으면 가로 스크롤을 허용한다.

컬럼 묶음 원칙:
- 컬럼 수를 늘리기보다 관련 정보를 2줄 셀로 묶는다.
- 임의의 정보가 주어지면 먼저 각 항목의 업무 의미, 비교 빈도, 편집 단위, 표시 길이를 판단한 뒤 묶는다.
- 같은 사람/대상에 속한 식별 정보는 묶는다. 예: 이름, 나이, 성별 / 회원명, 관리번호 / 의뢰인, 연락처
- 같은 연락 경로 또는 위치 정보는 묶는다. 예: 이메일, 휴대폰 / 주소, 국가 / 지역, 지점
- 같은 상태 흐름에 속한 값은 묶는다. 예: 결제상태, 결제수단 / 매칭상태, 배정상태 / 진행상태, 마감일
- 같은 시간 축에 있는 값은 묶는다. 예: 가입일, 채널 / 등록일, 수정일 / 결제일, 환불일 / 최근 로그인, 체류시간
- 같은 금액 계산에 쓰이는 값은 묶는다. 예: 금액, 수수료 / 결제금액, 환불금액 / 정산액, 미정산액
- 같은 상품/서비스 분류 값은 묶는다. 예: 상품, 서비스 / 유형, 카테고리 / 언어, 분야
- 운영자가 독립적으로 정렬, 필터, 일괄 선택해야 하는 핵심 값은 무리하게 묶지 않는다. 예: 상태, 담당자, 결제일, 금액, 등급이 화면의 주요 비교 기준이면 별도 컬럼으로 둔다.
- 한 셀에 3개 이상 성격이 다른 값을 넣지 않는다. 2줄을 기본으로 하고, 보조값은 작은 badge 또는 muted text로만 제한한다.
- 값 길이가 긴 항목끼리는 묶지 않는다. 긴 주소, 긴 상품명, 긴 메모는 다른 긴 텍스트와 합치지 않고 ellipsis 또는 Drawer 상세로 보낸다.
- 판단이 애매하면 `주요값 / 보조값` 구조로 둔다. 1줄에는 식별·비교에 필요한 주요값, 2줄에는 상태·출처·날짜 같은 보조값을 배치한다.
- 이름 / 나이·성별
- 이메일 / 휴대폰
- 주소 / 국가
- 회원유형 / 구독
- 가입일 / 채널
- 최근 로그인 / 체류시간
- 전문가 / 레벨·평점
- 매칭 / 상태
- 결제상태 / 수단
- 상품 / 서비스
- 금액 / 수수료
- 등록일 / 수정일
- 상태 / 담당자

상태/유형 표시:
- 반복되는 상태와 유형은 텍스트 색상만으로 표시하지 않는다.
- `.status.ok`, `.status.wait`, `.status.fail`, `.status.info`를 사용한다.
- 상태 chip은 상태 점, 연한 배경, border, 텍스트를 함께 가진다.
- 결제수단, 서비스유형, 구독등급, 우선순위는 `badge` 또는 보조 chip으로 표시한다.
- 자세한 기준은 `components/상태_유형_칩_기준.md`를 따른다.

행 상호작용:
- 행 전체를 클릭 대상으로 만든다.
- 선택된 행은 연한 primary 배경과 좌측 primary 인디케이터로 표시한다.
- 체크박스 선택 컬럼과 관리번호/ID 컬럼은 좌측 고정처럼 보이게 한다.
- 포커스 가능한 행은 focus-visible outline을 제공한다.

하단 일괄 액션:
- 선택 건수, 선택정보내기, 이메일/SMS 전송, 상태변경, 엑셀 내보내기, 페이지네이션을 둔다.
- 하단 액션은 table card 안쪽 하단에 붙인다.

시각 스타일:
- 배경은 흰색과 밝은 회색 중심이다.
- 테이블 헤더는 `#eceef2` 계열이다.
- 셀 경계는 `#eee` 또는 `#ddd` 얇은 라인이다.
- primary는 active 탭, primary button, 선택 행, 핵심 상태에만 사용한다.
- radius는 4~6px 수준만 사용한다. chip/status는 pill radius를 허용한다.

반응형:
- 데스크톱 BO 화면을 우선한다.
- 모바일에서도 테이블을 카드로 완전히 바꾸지 않는다.
- 핵심 컬럼을 남기고 가로 스크롤을 허용한다.

금지:
- 사용자용 hero, 큰 카드, 넓은 여백, 장식 이미지로 BO 화면을 채우지 않는다.
- 날짜를 `YYYY-MM-DD`로 쓰지 않는다.
- 하위 탭 카운트를 단독 배지만으로 표현하지 않는다.
- 이름, 나이, 성별처럼 강하게 연결된 정보를 불필요하게 별도 컬럼으로 분리하지 않는다.
- 상태값을 글자색만 다르게 해서 끝내지 않는다.
