# 디자인 시스템 ver3 밀도 강화 및 Drawer 전환 프롬프트

아래 프롬프트는 현재 `index.html`, `layout.css`, `common.js`, `dense-drawer.css`, `dense-drawer.js`, `docs/디자인시스템.md`를 기준으로 디자인 시스템의 마크업 밀도를 높이고, BO 회원 테이블 상세 편집 방식을 우측 Drawer로 유지/확장하기 위한 작업 지시문입니다.

현재 변경 방향:
- 이 버전에서 가장 중요한 원칙은 밀도 강화다. 같은 정보를 더 낮은 높이와 더 적은 빈칸으로 읽을 수 있으면 그 방식을 우선한다.
- `index.html` 자체를 최종 기준 문서로 업데이트하며, 별도 실험 파일에만 남기지 않는다.
- 검색/필터/체크박스/옵션 그룹은 항목이 적을 때 불필요하게 양끝으로 벌어지지 않아야 한다.
- 항목 수가 적으면 `justify-start`처럼 앞에서부터 자연스럽게 붙고, 항목이 많아지면 줄바꿈과 가용 폭을 사용해 화면을 최대한 채운다.
- 검색바에는 다수 옵션을 넣어 실제로 많은 조건이 들어왔을 때도 레이아웃이 깨지지 않는지 테스트한다.
- 회원목록 상세 편집은 기존의 행 하단 아코디언 방식이 아니라 우측 Drawer 방식으로 바꾼다.
- Drawer는 목록 맥락을 유지하면서도 테이블 높이를 늘리지 않는 BO 상세 패턴으로 정의한다.
- 참고예시는 한두 개가 아니라 검색/필터, 카드, 목록, 체크박스, 태그, 타임라인, sparse 정보, attribute overflow까지 충분히 많이 둔다.

## 1. 분석 프롬프트

```text
너는 시니어 프론트엔드 엔지니어이자 디자인 시스템 아키텍트다.

현재 레포지토리의 휴텍씨 디자인 시스템을 기준으로, 마크업 밀도를 높이는 새 버전을 설계해라. 이 단계에서는 파일을 수정하지 말고, 기존 구조와 수정 지점을 분석한 뒤 실행 계획만 제시해라.

우선 확인할 기준 파일:
- index.html
- layout.css
- common.js
- dense-drawer.css
- dense-drawer.js
- docs/디자인시스템.md
- docs/디자인시스템_고도화_프롬프트.md

핵심 목표:
1. 밀도 강화가 최우선이다. 정보가 읽히는 범위 안에서는 카드 높이, 행 높이, gap, padding을 줄이고 한 줄 표현을 우선한다.
2. 검색/필터/체크박스/옵션 그룹이 항목 수가 적을 때 `justify-content: space-between`처럼 양끝으로 벌어지지 않게 한다.
3. 항목 수가 적으면 좌측부터 조밀하게 배치하고, 항목 수가 많아지면 줄바꿈 또는 그리드 흐름으로 가용 행을 최대한 사용하게 한다.
4. 컨테이너가 넓어도 각 옵션이 불필요하게 늘어나지 않도록 `width: auto`, `flex: 0 0 auto`, `max-width`, `minmax`, `auto-fit/auto-fill` 사용 여부를 판단한다.
5. 검색바에는 실제 운영 화면처럼 많은 조건을 넣어 테스트한다. 예: 검색어, 회원유형, 권한, 상태, 등급, 가입경로, 마케팅 수신, 이메일 인증, 휴면 여부, 최근 로그인, 가입일 기간, 담당자, 정렬, 활성 필터 태그.
6. 활성 필터 태그와 `필터 전체 해제`는 항목이 적을 때 오른쪽 끝으로 밀리지 않아야 한다. 필요하면 별도 action row를 만들되 기본은 flow 안에서 자연스럽게 배치한다.
7. 회원목록 상세 편집은 기존 행 하단 아코디언을 폐기하고 우측 Drawer로 전환한다.
8. Drawer는 `role="dialog"`, `aria-modal` 적용 여부를 명확히 판단한다. 목록을 계속 조작할 수 없는 오버레이 Drawer라면 modal dialog로, 목록과 병행 조회하는 고정 side panel이면 complementary 영역으로 설계한다.
9. 기존 radius 0 정책은 유지한다. 칩, 스위치, 서브탭 예외도 새 밀도 규칙과 충돌하지 않게 정리한다.
10. 기존 클래스명과 문서 톤을 최대한 유지하되, 새 패턴이 필요하면 `dense-*`, `drawer-*`, `member-drawer-*`처럼 역할이 분명한 이름을 제안한다.
11. 접근성은 focus 이동, ESC 닫기, 닫은 뒤 트리거 버튼으로 focus 복귀, Drawer 제목 연결, 저장/삭제 버튼의 위험도 구분을 기준으로 점검한다.

분석해야 할 기존 지점:
- `layout.css`의 `.filter-bar`, `.filter-search`, `.filter-group`, `.filter-active`, `.filter-clear`
- `index.html`의 `검색·필터링 패턴` 예시
- `index.html`의 `회원목록 + 우측 Dense Drawer` 예시
- `common.js`의 `[data-filter-bar]` 상호작용
- `dense-drawer.js`의 `[data-drawer-open]`, `[data-drawer-form]`, `[data-drawer-close]` 상호작용
- `docs/디자인시스템.md`의 `4.6 Table` 회원목록 규칙

출력 형식:

1. 현재 레이아웃 문제 요약
   - 항목이 적을 때 불필요하게 벌어지는 지점
   - 항목이 많을 때 줄바꿈, overflow, 버튼 밀림 가능성이 있는 지점
   - 문서 규칙과 실제 예시가 충돌하는 지점

2. 밀도 강화 레이아웃 원칙
   - `space-between`을 피해야 할 컴포넌트
   - `justify-start`, `align-content: start`, `flex-wrap`, `gap` 기준
   - 검색 입력만 늘어나고 옵션/버튼은 내용 폭을 유지하는 기준
   - 많은 옵션이 있을 때 행을 최대한 쓰는 기준
   - 모바일에서 스택 처리할 기준

3. 검색바 다옵션 테스트 설계
   - 포함할 옵션 그룹 목록
   - 각 그룹의 단일 선택/다중 선택 여부
   - 활성 필터 태그 생성 방식
   - `필터 전체 해제` 버튼 위치
   - 데스크톱/태블릿/모바일 검증 포인트

4. 회원 Drawer 전환 설계
   - 기존 하단 아코디언에서 제거할 마크업과 JS
   - 새 Drawer DOM 구조
   - 열기/닫기/focus/ESC/저장/삭제 상호작용
   - Drawer 폭, 위치, 스크롤, 모바일 전체 화면 처리
   - 테이블 행 선택 상태와 Drawer 제목/내용 동기화 방식
   - 최근 활동, 처리 이력, 인증 이력처럼 짧은 이벤트성 정보는 항목 내부는 한 줄, 목록 전체는 2~4단으로 펼치는 기준
   - 기본 정보 입력은 `라벨 · 입력필드`를 가능한 한 한 줄로 배치하는 기준

5. 접근성 및 QA
   - 키보드 이동
   - focus-visible
   - Drawer 제목과 닫기 버튼 label
   - 삭제 확인 흐름
   - 텍스트 겹침, 줄바꿈, 가로 스크롤 점검

6. 예상 변경 파일
   - 수정할 파일과 이유
   - 새로 추가할 수 있는 class/data attribute
   - 문서에서 바꿔야 할 규칙

7. 구현 순서
   - 작은 작업 단위로 나누고, 각 단계마다 검증 방법을 적어라.

주의:
- 이 응답에서는 실제 파일을 수정하지 마라.
- 기존 디자인 시스템의 radius 0 정책과 BO 고밀도 정보 구조는 유지해라.
- 단순히 여백만 줄이지 말고, 항목 수에 따라 자연스럽게 차는 레이아웃 규칙을 만들어라.
- 회원 상세는 하단 아코디언이 아니라 우측 Drawer를 기준으로 설명해라.
```

## 2. 후속 구현 프롬프트

분석 결과를 검토한 뒤 실제 코드를 수정할 때는 아래 프롬프트를 사용한다.

```text
앞서 만든 디자인 시스템 밀도 강화 및 Drawer 전환 분석 결과를 기준으로 실제 코드와 문서를 수정해라.

작업 기준:
- `index.html`을 기준 문서로 유지한다.
- 기존 섹션과 컴포넌트 예시는 임의로 삭제하지 않는다.
- 변경 범위는 검색/필터 패턴, 회원 테이블 상세 패턴, 관련 문서 규칙에 집중한다.
- 검색/필터/체크박스/옵션 그룹은 항목이 적을 때 좌측부터 조밀하게 배치한다.
- 항목이 많아지면 줄바꿈과 가용 폭을 사용해 행을 최대한 채운다.
- `justify-content: space-between`은 헤더, 푸터, 명확한 양끝 액션 구조처럼 필요한 곳에만 남긴다.
- 필터 옵션, 체크박스 옵션, 활성 필터 태그는 `justify-start` 기반 flow로 정리한다.
- 검색 입력은 필요한 경우에만 늘어나고, 칩/버튼은 내용 폭을 유지한다.
- 검색바 예시는 많은 옵션을 포함한 고밀도 테스트 케이스로 확장한다.
- 회원목록의 상세 편집은 우측 Drawer 기준을 유지하고, 하단 아코디언 패턴이 다시 들어오지 않게 한다.
- Drawer는 데스크톱에서 우측 고정 패널, 모바일에서 화면 폭을 사용하는 패널로 동작하게 한다.
- Drawer 열기 버튼은 `aria-expanded`와 대상 연결을 유지하고, 닫기 후 focus를 원래 버튼으로 되돌린다.
- ESC로 닫기, backdrop 또는 닫기 버튼으로 닫기, 저장/삭제/닫기 액션을 구현한다.
- 삭제는 위험 액션이므로 기존 danger 버튼 스타일을 사용하고 확인 흐름을 남긴다.
- `docs/디자인시스템.md`의 회원목록 규칙을 하단 아코디언이 아닌 우측 Drawer 기준으로 갱신한다.
- `index.html`에는 밀도 강화 원칙 카드, 전체 검색 옵션, 카드/목록 케이스, 체크박스 flow, 2~4단 로그, 회원 상세 Drawer 샘플을 모두 반영한다.
- 참고예시는 실제 운영에서 판단해야 하는 양을 보여줄 수 있도록 충분히 많이 만든다. 적은 정보, 많은 정보, 많은 옵션, 많은 속성, 짧은 이벤트 로그를 모두 포함한다.
- 버튼/링크 컴포넌트처럼 variant가 많은 기본 UI는 한 줄씩 반복 나열하지 않는다.
- 버튼은 variant x size 매트릭스로 비교하고, 링크는 테이블 값/보조/위험처럼 역할별 조밀 목록으로 정리한다.
- 동일 역할의 버튼/링크 예시가 `UI별 보기`와 상세 섹션에 중복될 때는 상세 섹션에서 더 압축된 기준표를 제공하고, 카드 미리보기는 최소 샘플만 둔다.
- 변경 후 HTML/CSS/JS를 확인하고 가능하면 브라우저에서 데스크톱/모바일 시각 검증을 수행한다.

구현할 검색바 테스트 옵션 예시:
- 검색어 입력
- 회원유형: 전체, 일반회원, 강사, 관리자, 기업회원
- 권한: 읽기, 작성, 승인, 정산, 시스템관리
- 상태: 활성, 휴면, 정지, 탈퇴대기
- 등급: Basic, Silver, Gold, VIP
- 가입경로: 홈페이지, 관리자등록, 캠페인, 제휴사
- 수신동의: 이메일, SMS, 카카오 알림
- 인증상태: 이메일 인증, 휴대폰 인증, 미인증
- 기간: 가입일 시작/종료, 최근 로그인 시작/종료
- 담당자: 전체, 박민수, 정유진, 오지훈
- 정렬: 최신 가입순, 최근 로그인순, 이름순, 등급순

검증:
- 검색바 옵션이 적을 때 양끝으로 벌어지지 않는지 확인한다.
- 옵션이 많을 때 한 줄에 억지로 압축되지 않고 여러 행을 사용하면서 정렬이 유지되는지 확인한다.
- 활성 필터 태그가 많아져도 `필터 전체 해제`가 어색하게 오른쪽 끝으로 밀리지 않는지 확인한다.
- 회원 상세를 열었을 때 테이블 하단에 새 행이 생기지 않고 우측 Drawer가 열린다.
- Drawer를 닫으면 선택 행 상태와 버튼 `aria-expanded`가 원복된다.
- 키보드로 Drawer를 열고 닫을 수 있고 focus가 복귀한다.
- 모바일에서 Drawer와 테이블이 겹쳐 읽기 어려운 상태가 되지 않는다.
- 최근 활동 타임라인은 목록 전체를 1열로 고정하지 않는다. Drawer 폭이 허용하면 2~4단 그리드로 펼친다.
- 각 타임라인 항목 내부만 `시간 · 이벤트명 · 처리자/상태` 한 줄로 유지한다.
- "시험 접수 결제 완료", "이메일 인증 완료"처럼 짧은 이벤트는 별도 카드나 여러 줄 설명으로 늘리지 않는다.
- 최근 활동/인증/처리 이력은 카드형 여백보다 로그형 밀도를 우선한다. 행 높이는 26~30px 수준을 기준으로 잡고, border box 카드 반복보다 얇은 구분선 반복을 우선한다.
- 타임라인 gap과 padding은 최소화한다. 예: `gap: 3~6px`, `padding-block: 3~4px`, `font-size: 11~12px`.
- 이벤트명이 길어질 때만 핵심 이벤트명을 한 줄 말줄임 처리하고, 상세 설명은 title, tooltip, 상세 로그 또는 별도 펼침 영역으로 분리한다.
- Drawer 기본 정보 form은 `이름` 라벨과 `이름 입력필드`처럼 라벨/필드를 다른 열 카드로 분리하지 않는다.
- 기본 정보 입력은 `label 72px + field 1fr` 같은 한 줄 row를 우선 적용한다.
- 이름, 권한, 연락처, 이메일, 상태, 등급, 담당자처럼 짧은 필드는 모두 한 줄 row로 유지한다.
- 관리 메모, 긴 주소, 긴 사유처럼 장문 입력만 예외적으로 같은 row 안에서 textarea 높이를 키우거나 세로 배치를 허용한다.

마지막에 수정한 파일, 변경 이유, 검증 결과를 요약해라.
```

## 3. 마크업 기준 스니펫 프롬프트

구현자가 레이아웃 규칙을 헷갈리지 않도록 아래 기준을 함께 전달한다.

```text
검색/필터처럼 항목 수가 유동적인 영역은 기본적으로 flow layout으로 작성한다.

권장 구조:
- 검색 입력 영역은 `.dense-search-main`
- 옵션 묶음은 `.dense-filter-flow`
- 각 옵션 그룹은 `.dense-filter-group`
- 활성 태그 영역은 `.dense-active-filters`
- 우측 고정 액션이 꼭 필요할 때만 `.dense-filter-actions`를 별도 행으로 둔다.

현재 BO 회원 검색조건 CSS 기준:
- `.dense-filter-flow`는 `display: flex`로 둔다.
- 역할 구분 wrapper인 `.dense-filter-role-group`은 `display: contents`로 구조만 유지한다.
- `.dense-filter-role-body`는 컬럼 구분선 역할을 하며 `height: 100%`, `border-right: 1px solid var(--line-soft)`를 사용한다.
- `.dense-filter-group` 내부는 grid를 한 번 더 만들지 않고 `display: flex`로 둔다.
- 조건명은 `legend { flex: 0 0 58px; }`, 옵션 묶음은 `.dense-filter-options { flex: 1 1 auto; display: flex; flex-wrap: wrap; }`로 맞춘다.
- 기간설정처럼 긴 조건은 `.filter-date-group .dense-filter-options { flex-wrap: nowrap; overflow-x: auto; }`로 한 줄 유지한다.

금지 또는 제한:
- 옵션 그룹 전체에 `justify-content: space-between`을 기본으로 주지 않는다.
- 버튼 하나를 오른쪽 끝으로 보내기 위해 `margin-left: auto`를 남발하지 않는다.
- 체크박스가 2~3개뿐인데 컨테이너 전체 폭에 균등 분산하지 않는다.
- 칩이나 체크박스 라벨을 강제로 같은 폭으로 늘리지 않는다.

회원 상세:
- 테이블 행 아래 `<tr class="member-detail-row">`를 펼치는 방식은 새 버전에서 사용하지 않는다.
- `data-drawer-open` 버튼은 우측 Drawer를 여는 트리거가 된다.
- Drawer 안에는 회원 기본 정보, 권한/상태/등급, 연락처, 메모, 위험 액션을 배치한다.
- Drawer는 목록을 가리지 않는 폭을 우선하되, 좁은 화면에서는 전체 폭 패널로 전환한다.
- Drawer의 활동/이력 리스트는 우선 2~4단 밀도형으로 만든다.
- 활동/이력 항목 하나의 내부 구조는 `시간`, `이벤트명`, `처리자 또는 상태` 3요소를 한 줄로 둔다.
- 활동/이력 컨테이너는 `grid-template-columns: repeat(auto-fit, minmax(200px, 1fr))` 같은 방식으로 2~4단을 만든다.
- 이벤트명은 `minmax(0, 1fr)`로 남은 폭을 쓰고 `white-space: nowrap`, `overflow: hidden`, `text-overflow: ellipsis`를 적용한다.
- 처리자/상태는 `max-content`로 유지하고, 긴 설명은 같은 줄에 억지로 모두 넣지 않는다.
- 모바일에서도 가능하면 2단을 유지하고, 아주 좁은 화면에서만 1단으로 내린다.
- Drawer 기본 정보 form은 `.drawer-inline-field` 같은 한 줄 row 패턴을 사용한다.
- `.drawer-inline-field`는 `grid-template-columns: 72px minmax(0, 1fr)`처럼 라벨 폭만 고정하고 입력필드가 남은 폭을 사용하게 한다.
- 라벨과 입력필드를 각각 별도 카드/열로 나누지 않는다.
- 한 줄 row가 많아질 때는 row 자체를 `auto-fit/minmax` 그리드로 배치해 Drawer 폭을 채운다.
```
