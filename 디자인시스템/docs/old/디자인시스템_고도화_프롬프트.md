# 디자인 시스템 ver2 통합 고도화 프롬프트

아래 프롬프트는 현재 `index.html`을 기준 문서로 삼아, 기존 `index.html`의 디자인 시스템을 보존하면서 시험 랜딩페이지와 이후 추가될 페이지/컴포넌트까지 통합 적용하기 위한 작업 지시문입니다.

현재 상태 기준:
- `index.html`은 기존 `index.html`의 섹션과 예시를 유지하면서 `고도화` 탭, 게시판 예시, radius 0 정책, 컴포넌트 보완 체크리스트를 추가한 버전이다.
- `layout.css`에는 공통 토큰, 내비게이션, 패널, 컴포넌트 스타일이 있고, `index.html`은 일부 보완 스타일을 문서 내부 `<style>`로 추가하고 있다.
- `common.js`는 컴포넌트 필터, 탭, 토스트, 모달, Rules 미리보기, 레이아웃 선택 예시 등 문서 상호작용을 담당한다.
- 시험 랜딩페이지 관련 실물 폴더는 현재 저장소 루트에서 확인되지 않았으므로, 프롬프트 실행 시 실제 경로가 있으면 우선 확인하고 없으면 `index.html`, `docs/시험사이트_분석/*`, `resource/시험사이트_분석/*`, `docs/DB_정리/*`를 근거로 적용 범위를 판단한다.

## 1. 분석 및 통합 설계 프롬프트

```text
너는 시니어 프론트엔드 엔지니어이자 디자인 시스템 아키텍트다.

현재 레포지토리의 휴텍씨 디자인 시스템을 `index.html` 기준으로 고도화하고, 시험 랜딩페이지까지 통합 적용할 수 있는 구조를 설계해라. 이 단계에서는 파일을 수정하지 말고, 코드베이스와 문서를 분석한 뒤 실행 계획만 제시해라.

우선 확인할 기준 파일:
- index.html
- index.html
- layout.css
- common.js
- docs/디자인시스템.md
- docs/시험사이트_분석/*
- resource/시험사이트_분석/*
- docs/DB_정리/*
- 시험_랜딩페이지/layout_redesign 또는 시험 랜딩페이지 관련 실제 경로가 있으면 해당 폴더 전체

핵심 판단 기준:
1. `index.html`을 앞으로의 기준 문서로 삼되, 기존 `index.html`에 있던 섹션/컴포넌트/문서 예시가 유실되지 않았는지 확인한다.
2. `layout.css`의 공통 토큰과 `index.html` 내부 보완 스타일 중 어떤 항목을 공통 CSS로 승격해야 하는지 구분한다.
3. radius 0 정책을 기본값으로 유지하되, 게시판 하위 서브탭처럼 pill 형태가 필요한 예외를 명확히 문서화한다.
4. 관리자 화면과 사용자 화면을 분리해서 본다. 관리자 화면은 검색/필터/대량 테이블/엑셀/상태 관리 중심이고, 사용자 화면과 랜딩페이지는 서비스 소개/신청 흐름/FAQ/CTA/신뢰 요소 중심이다.
5. 시험 랜딩페이지는 별도 디자인을 새로 만들기보다 `index.html`의 Foundation, Components, Pages 패턴으로 조립 가능한지 먼저 판단한다.
6. 시험, 번역, 문서, 매칭, 교육, 서비스 홈페이지 등 여러 도메인에 공통 적용 가능한 패턴과 도메인별 변형이 필요한 영역을 나눈다.
7. 추가될 가능성이 높은 컴포넌트를 미리 고려하되, 실제 구현 전에는 토큰/variant/state/API 수준까지만 정리한다.
8. 접근성은 색상 대비, focus 상태, disabled 상태, 키보드 이동, 의미 있는 HTML 구조, aria 속성, modal focus 복귀를 기준으로 점검한다.
9. 반응형은 단순 축소가 아니라 정보 밀도와 업무 흐름이 유지되는지 기준으로 확인한다.
10. 불필요한 대규모 리팩터링을 제안하지 말고, 기존 클래스명과 문서 톤을 최대한 유지한다.

분석해야 할 기존 컴포넌트:
- Button: primary, secondary, ghost, danger, disabled, icon 포함 여부
- Link: 본문 링크, 외부 링크, CTA 링크
- Tabs: 기본 탭, 세그먼트 탭, pill 서브탭, nav category, section tabs
- Board/List: 공지사항형 게시판, 모바일 축약형 목록
- Form: input, select, textarea, checkbox, radio, search, date/range, file upload
- Header/GNB: 사용자 홈페이지형, 관리자형, 문서형
- Sidebar/LNB: 문서 목차, 관리자 아이콘 메뉴, 카테고리 메뉴
- Footer: 홈페이지형, 문서형, 간소형
- Table/Data grid: 관리자 대량 목록, 상태 컬럼, row action, horizontal scroll
- Feedback: toast, modal, popup, alert, confirmation
- Card/Panel: 정보 카드, 규칙 카드, 기능 카드, 요약 패널
- Badge/Chip/Status: 상태값, 카테고리, 필터 토큰
- Pagination: 목록/게시판/관리자 테이블 페이지 이동

추가 가능성이 높은 컴포넌트 후보:
- Hero: 시험/서비스 랜딩페이지 첫 화면, 핵심 CTA, 일정/접수 상태 표시
- CTA band: 신청, 상담, 견적, 전문가 매칭 유도 영역
- Pricing/Fee table: 시험 응시료, 서비스 비용, 환불/정산 안내
- Schedule/Timeline: 시험 접수, 응시, 채점, 결과 발표 흐름
- Stepper/Process: 신청서 작성, 결제, 응시, 결과 확인 단계
- FAQ/Accordion: 시험 안내, 환불, 자격, 준비물, 문의
- Notice/Announcement: 공지, 긴급 안내, 점검 안내
- Empty/Loading/Error state: 데이터 없음, 처리 중, 실패 상태
- File upload/Attachment: 엑셀 업로드, 증빙서류, 시험 자료 업로드
- Filter bar/Advanced search: 관리자 조건 검색, 날짜/상태/분류 필터
- Bulk action toolbar: 선택 항목 일괄 처리, 엑셀 다운로드, 상태 변경
- Detail summary: 회원/시험/결제/결과를 한 화면에서 요약
- Stats/KPI card: 접수 수, 결제 수, 합격률, 처리 상태
- Breadcrumb: 관리자/문서/상세 화면 위치 표시
- Language pair selector: 번역/시험 도메인의 출발어/도착어 선택
- Exam card: 시험 종목, 등급, 접수 기간, 응시 가능 상태 표시
- Certificate/Result block: 합격/불합격, 점수, 수료증/자격증 안내

출력 형식:

1. 현재 구조 분석
   - `index.html`, `index.html`, `layout.css`, `common.js`의 역할
   - `index.html`에서 새로 추가된 고도화 요소
   - 기존 index와 비교해 보존/변경/추가된 항목

2. 디자인 토큰 분석
   - 색상, 타이포그래피, 간격, radius, shadow, z-index, 상태 색상
   - 공통 CSS로 승격할 항목
   - 문서 내부 스타일로 남겨도 되는 항목

3. 컴포넌트 현황
   - 이미 문서화된 컴포넌트
   - 상태/variant/size 정의가 부족한 컴포넌트
   - 시험 랜딩페이지 적용 시 바로 쓸 수 있는 컴포넌트
   - 추가 설계가 필요한 컴포넌트 후보

4. 시험 랜딩페이지 통합 적용 방향
   - 사용자 랜딩페이지에 필요한 섹션 구조
   - 관리자 시험 운영 화면과 분리해야 할 패턴
   - 공통으로 재사용할 토큰/컴포넌트
   - 도메인별로 변형해야 할 콘텐츠/상태값

5. 접근성 및 반응형 점검
   - focus, aria, 키보드 이동, disabled, modal, toast
   - 모바일에서 숨김/축약/스택 처리해야 할 요소
   - 테이블과 게시판의 모바일 대응

6. 우선순위별 실행 계획
   - 1단계: 문서와 기준 정리, 위험 낮은 CSS 보완
   - 2단계: 공통 컴포넌트 variant/state 표준화
   - 3단계: 시험 랜딩페이지 섹션 패턴 추가
   - 4단계: 관리자 대량 테이블/필터/업로드 패턴 정리
   - 5단계: 접근성, 반응형, 시각 QA

7. 예상 변경 파일
   - 변경 가능성이 높은 파일
   - 새로 만들면 좋은 문서 또는 예시 파일
   - 변경 전 반드시 확인해야 할 파일

8. 다음 작업 제안
   - 사용자가 바로 승인할 수 있는 작은 작업 단위로 나눠 제시해라.
   - 각 작업마다 기대 효과와 위험도를 짧게 설명해라.

주의:
- 이 응답에서는 실제 파일을 수정하지 마라.
- 구체적인 파일명과 근거를 들어라.
- 모호한 개선안보다 이 레포에서 바로 실행 가능한 개선안을 우선해라.
- `index.html`을 기준으로 판단하되, 기존 `index.html`에서 유실된 내용이 있으면 반드시 지적해라.
- 시험 랜딩페이지 실제 경로가 없으면 없다고 명시하고, 현재 문서와 DB 자료 기준의 추정 적용안으로 분리해라.
```

## 2. 후속 구현 프롬프트

분석 결과를 검토한 뒤 실제 코드를 수정할 때는 아래 프롬프트를 사용한다.

```text
앞서 만든 디자인 시스템 ver2 통합 고도화 분석 결과를 기준으로, 1단계 작업부터 실제 코드와 문서를 수정해라.

작업 기준:
- `index.html`을 기준 문서로 유지한다.
- 기존 `index.html`의 내용은 임의로 삭제하거나 덮어쓰지 않는다.
- 변경 범위를 작게 유지한다.
- 기존 클래스명, CSS 변수명, 문서 톤을 최대한 유지한다.
- `index.html` 내부에 임시로 들어간 보완 스타일 중 반복 사용 가치가 있는 항목만 `layout.css`로 옮긴다.
- radius 0 정책을 기본값으로 유지하고, 예외는 명시적으로 제한한다.
- 버튼, 링크, 탭, 게시판, 폼, 테이블, 피드백 컴포넌트의 variant/size/state 문서화를 우선한다.
- 시험 랜딩페이지에 필요한 Hero, CTA, Schedule, FAQ, Pricing/Fee, Stepper, Notice 컴포넌트는 기존 토큰과 패턴으로 조립한다.
- 관리자 대량 운영 화면에 필요한 Filter bar, Bulk action toolbar, Data grid, File upload, Status badge는 사용자 랜딩페이지 컴포넌트와 분리해서 정리한다.
- 변경 후에는 영향받는 HTML/CSS/JS를 확인하고, 가능하면 브라우저에서 시각적으로 검증한다.
- 수정한 파일, 변경 이유, 검증 결과를 마지막에 요약한다.
```

## 3. 시험 랜딩페이지 제작 프롬프트

시험 랜딩페이지를 실제 화면으로 만들 때는 아래 프롬프트를 사용한다.

```text
`index.html`과 `layout.css`의 디자인 시스템을 기준으로 시험 랜딩페이지 화면을 제작해라.

화면 목적:
- 사용자가 시험의 성격, 대상, 일정, 접수 방식, 응시 흐름, 비용, FAQ를 빠르게 이해하고 신청 또는 문의로 이동하게 한다.
- 관리자 대량 관리 화면처럼 보이지 않게 한다. 랜딩페이지는 사용자용 서비스 소개와 신청 전환 중심으로 구성한다.

필수 섹션:
1. Header/GNB
2. Hero: 시험명, 핵심 설명, 접수 상태, 주요 CTA
3. 시험 개요: 대상, 등급, 과목, 응시 방식
4. 일정/절차: 접수, 결제, 응시, 채점, 결과 발표
5. 비용/환불 또는 안내 표
6. 준비물/유의사항
7. FAQ
8. 최종 CTA
9. Footer

디자인 원칙:
- 기존 토큰과 컴포넌트를 우선 사용한다.
- radius는 기본 0으로 유지한다.
- CTA는 명확하지만 과도한 장식은 피한다.
- 일정, 상태, 비용처럼 비교가 필요한 정보는 표나 리스트로 정리한다.
- 모바일에서는 Hero CTA, 일정, FAQ가 먼저 읽히도록 스택 순서를 조정한다.
- 접근성 기준에 맞게 heading 구조, 버튼/링크 역할, focus 상태를 유지한다.

검증:
- 데스크톱과 모바일에서 텍스트가 겹치지 않는지 확인한다.
- CTA, FAQ, 탭/아코디언이 있으면 키보드로 조작 가능한지 확인한다.
- 기존 `index.html`의 컴포넌트 규칙과 어긋나는 임의 색상/간격/폰트 크기를 쓰지 않는다.
```
