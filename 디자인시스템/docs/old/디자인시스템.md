# 휴텍씨 디자인 시스템

기준 파일: `index.html`, `layout.css`, `common.js`, `dense-drawer.css`, `dense-drawer.js`, `table_design/*`

이 문서는 현재 `index.html`에 구현된 디자인 시스템 화면을 기준으로, 이후 시험/번역/서비스 홈페이지를 만들 때 반복 적용할 공통 규칙을 정리한 문서입니다.

## 1. 시스템 원칙

### 1.1 공통 틀 우선

홈페이지와 업무 화면은 개별 페이지를 바로 만들기보다 다음 공통 틀을 먼저 고정합니다.

- Header: 로고, 주요 메뉴, 사용자/알림 영역
- Sidebar 또는 Rail: 업무별 보조 탐색
- Content: 검색, 필터, 목록, 상세, 작성 영역
- Footer: 약관, 개인정보, SNS, 저작권

개별 홈페이지는 이 틀 위에 서비스 소개, 기능, 실적, 전문가 신청, 견적 의뢰, FAQ, 회사 소개를 조합합니다.

### 1.2 공통과 개별 분리

공통 UI와 업무 콘텐츠를 분리합니다.

- 공통 UI: 버튼, 입력, 탭, 칩, 상태 pill, 테이블, 토스트, 모달, 페이지네이션
- 개별 콘텐츠: 시험 종목, 번역 분야, 고전 번역 설명, 마케팅 문구, 신청서 항목

개별 홈페이지를 만들 때는 색상/문구/이미지/섹션 순서를 바꾸더라도 공통 UI class와 토큰은 유지합니다.

### 1.3 홈페이지 목차 원칙

첫 화면은 회사 소개보다 서비스 소개를 먼저 보여줍니다.

권장 순서:

1. 서비스 소개 / Hero
2. 기능, 분야, 현황, 기술 설명
3. 실적
4. 전문가 신청
5. 견적 의뢰
6. FAQ, 상담, 신청 흐름
7. 회사 소개

고전 번역처럼 독립 브랜드 성격이 강한 홈페이지는 기존 번역 사이트 틀을 재사용하되, 별도 색상 구조와 현황/기술 설명을 강화합니다.

### 2.2 간격, 곡률, 높이

| 분류 | 값 | 사용처 |
|---|---|---|
| Spacing | `2, 4, 6, 8, 10, 12, 16, 20, 24, 30px` | 내부 여백, gap |
| Radius | 기본 `0px`, 예외 `20px` | 기본 요소는 0, 스위치와 칩만 pill radius 유지 |
| Height | `24, 32, 40, 60px` | 소형 버튼, 기본 입력/버튼, 사이드 메뉴, GNB/Footer |

### 2.3 폰트 크기

관리자 시스템은 정보 밀도가 높으므로 작은 글자 기준을 사용합니다.

| 영역 | 기준 |
|---|---:|
| BO 기본 본문 | `8px` |
| BO 테이블 헤더 | `8px` |
| BO 테이블 셀 | `8px` |
| BO 보조 정보 | `8px` |
| BO 페이지 제목/상단바 | 화면 위계상 필요한 경우만 `12~14px` |

사용자 디자인 시스템은 관리자 기준보다 약 `+2px` 크게 잡습니다.

| 영역 | 기준 |
|---|---:|
| 제목 | `14~15px` |
| 표 제목 | `13~14px` |
| 내용 | `10~12px` |

권장 토큰:

```css

--ds-admin-table-title-font-size: 8px;
--ds-admin-table-cell-font-size: 8px;
--ds-admin-caption-font-size: 8px;

--ds-user-title-font-size: 15px;

--ds-user-body-font-size: 12px;
```

## 3. 접근성 기준

### 3.1 명도 대비

버튼, 칩, 상태 pill, badge, tag를 만들 때는 텍스트와 배경 또는 주요 경계의 명도 대비를 최소 `3:1` 이상으로 유지합니다.

현재 시스템에서 지켜야 할 조합:

- 연한 파란 배경 위 텍스트: `--ds-color-brand-primary-strong` 사용
- 컬러 버튼 위 텍스트: `--ds-color-text-inverse` 사용
- 비활성 pill: `#f2f2f2` 배경 위 `#555555` 사용
- 성공/경고/오류 텍스트: 일반 accent 색을 그대로 쓰지 말고 `*-strong` 토큰 사용

주의:

- `#0088ff` 텍스트를 `#d7e7fc` 위에 올리는 조합은 약하므로 피합니다.
- `#51b117`, `#f7b22c`를 흰 배경 위 작은 텍스트로 쓰지 않습니다. 아이콘/면 색상으로 쓰거나 strong 토큰을 함께 씁니다.
- 색상만으로 상태를 전달하지 말고 텍스트, 아이콘, `aria-pressed`, `aria-selected`, `role="switch"` 같은 상태 정보도 함께 제공합니다.

### 3.2 키보드와 포커스

- 버튼, 링크, 입력, 탭은 `:focus-visible` 스타일을 유지합니다.
- 탭은 `role="tablist"`, `role="tab"`, `aria-selected`를 사용합니다.
- 토글은 `role="switch"` 또는 `aria-pressed`를 사용합니다.
- 토스트 영역은 `aria-live="polite"`와 `aria-atomic="true"`를 사용합니다.
- 모달은 `role="dialog"`, `aria-modal="true"`, 제목 연결 `aria-labelledby`를 사용합니다.

## 4. 컴포넌트 규칙

### 4.1 Button

기본 class는 `btn`입니다. 변형은 `btn--primary`, `btn--outline`, `btn--neutral`, `btn--secondary`, `btn--danger`를 사용합니다.

크기:

- `btn--sm`: 24px
- `btn--md`: 32px
- `btn--lg`: 40px

기존 호환을 위해 `small`, `large`, `outline`, `neutral`, `secondary`, `danger` 별칭도 유지하지만 새 페이지에서는 `btn--*` 형식을 우선 사용합니다.
버튼 예시는 한 줄씩 반복 나열하지 않고 variant와 size를 매트릭스로 비교합니다.

### 4.2 Link

본문 이동이나 테이블 내 강조 값은 버튼 대신 `text-link`를 사용합니다.

- 기본 링크: `text-link`
- 보조 링크: `text-link muted-link`
- 위험 링크: `text-link danger-link`

링크 예시는 테이블 값, 보조 링크, 위험 링크처럼 역할별 조밀 목록으로 정리합니다.

### 4.3 Tab

탭은 역할에 따라 class를 선택합니다.

- `tabs--underline`: 페이지 상단 주요 구분
- `tabs--box`: 상세 정보 패널
- `tabs--pill`: 상태/카테고리 필터
- `tabs--segmented`: 목록/캘린더/요약 같은 보기 전환

선택 상태는 `.active`와 `aria-selected="true"`를 함께 맞춥니다.

### 4.4 Chip, Tag, Status Pill

필터와 상태 표시는 다음 class를 사용합니다.

- 필터 칩: `filter-chip`
- 활성 필터 태그: `filter-tag`
- 규칙 필터 칩: `chip-button`
- 상태 표시: `state-pill`
- 비활성/검토중 상태: `state-pill off`
- 규칙 라벨: `rule-tag`

접근성 규칙:

- 활성 칩은 `aria-pressed="true"`를 사용합니다.
- 연한 배경 위 텍스트는 strong 토큰을 사용합니다.
- 상태는 색상만으로 구분하지 않고 "완료", "검토중", "고정", "준고정"처럼 텍스트를 같이 표시합니다.

### 4.5 Form

입력 기본 class는 `field`입니다.

- 기본 높이: 32px
- 라벨: `label`
- 상세 입력: `textarea.field`
- 도움말/오류 문구가 필요한 경우 `field-help`, `field-error` 패턴을 사용합니다.

로그인/회원가입은 `auth-panel`, `auth-form` 패턴을 사용합니다.

- 로그인은 이메일, 비밀번호, 로그인 유지, 비밀번호 찾기, 로그인 버튼으로 최소 구성합니다.
- 회원가입은 이름, 휴대폰, 이메일, 회원 유형, 비밀번호, 비밀번호 확인, 약관 동의를 한 화면에서 처리합니다.
- 필수 약관은 `(필수)` 문구와 `required` 속성을 함께 사용합니다.
- 로그인과 회원가입 화면은 카드처럼 과하게 장식하지 않고 입력 흐름이 바로 보이도록 구성합니다.
- 제출 후 피드백은 토스트로 표시하고, 실제 서비스에서는 서버 오류/검증 오류를 `field-error`로 각 필드 아래에 표시합니다.

### 4.6 Table

시험 목록은 `exam-table`을 사용합니다.

- 테이블은 `table-scroll` 안에 넣어 가로 스크롤을 허용합니다.
- 헤더는 12px 보조 텍스트, 본문은 14px 기준입니다.
- 목록 폭이 넓으면 모바일에서 억지로 줄이지 않고 가로 스크롤을 허용합니다.

회원목록처럼 한 행에서 많은 정보를 보고 바로 수정해야 하는 BO 화면은 `member-table` 패턴을 사용합니다.

- 목록 행에는 회원번호, 이름, 권한, 연락처, 이메일, 생년월일, 성별, 가입일, 최근 로그인, 상태, 등급, 담당자, 메모처럼 판단에 필요한 정보를 충분히 노출합니다.
- 행의 `상세` 버튼을 선택하면 테이블 하단에 행을 추가하지 않고 우측 Drawer를 엽니다.
- Drawer는 넓은 BO 상세 패널로 사용하며 기본 정보, 요약 카드, 권한, 최근 활동, 태그, 체크리스트, 많은 속성 목록을 같은 화면에서 조밀하게 보여줍니다.
- 기본 정보 입력은 라벨과 입력필드를 별도 카드/열로 나누지 않고 `라벨 72px + 입력필드 1fr` 형태의 한 줄 row로 배치합니다.
- 이름, 권한, 연락처, 이메일, 상태, 등급, 담당자처럼 짧은 필드는 모두 한 줄 row로 유지합니다.
- 관리 메모, 긴 주소, 긴 사유처럼 장문 입력만 예외적으로 같은 row 안에서 textarea 높이를 키웁니다.
- 최근 활동, 인증, 처리 이력은 목록 전체를 1열로 고정하지 않고 Drawer 폭이 허용하면 2~4단 그리드로 펼칩니다.
- 각 활동 항목 내부는 `시간 · 이벤트명 · 처리자/상태` 한 줄을 유지하고, 긴 이벤트명은 말줄임 처리합니다.
- Drawer 하단에는 `삭제`, `닫기`, `저장` 버튼을 둡니다.
- 삭제는 위험 액션이므로 실제 서비스에서는 확인 팝업을 한 번 더 연결합니다.
- 한 번에 하나의 Drawer만 열고, 선택 행은 배경으로 표시합니다.
- Drawer는 `role="dialog"`, 제목 연결, 닫기 버튼, ESC 닫기, 닫은 뒤 트리거 버튼으로 focus 복귀를 지원합니다.

### 4.6.1 BO 회원목록 inline Drawer

`table_design/관리자패널_전체회원.html`은 `index.html`의 "회원목록 + 우측 Dense Drawer" 패턴을 별도 BO 페이지에 적용한 현재 기준 예시입니다.

- 회원 검색조건은 `dense-filter-card` 안에서 `dense-filter-flow`를 사용하며, 현재 CSS 기준은 flex flow입니다.
- 검색조건 역할 wrapper인 `dense-filter-role-group`은 `display: contents`로 구조만 유지하고, `회원 기본`, `서비스/상태`, `운영/접근` 같은 그룹명은 화면에 노출하지 않습니다.
- `dense-filter-role-body`는 높이 100%와 우측 border로 컬럼 구분선을 만들며, 마지막 그룹의 border는 제거합니다.
- 개별 조건은 `dense-filter-group { display: flex; }`이고, `legend`는 `flex: 0 0 58px`, 옵션 묶음은 `.dense-filter-options { flex: 1 1 auto; display: flex; flex-wrap: wrap; }` 기준으로 맞춥니다.
- 기간설정은 `.filter-date-group .dense-filter-options`에서 `flex-wrap: nowrap; overflow-x: auto;`로 한 줄을 유지합니다.
- 테이블 행을 클릭하면 오버레이 모달이 아니라 목록 영역 오른쪽 컬럼에 inline Drawer가 붙어서 열립니다.
- 선택된 row는 연한 primary 배경과 좌측 primary 인디케이터로 표시합니다.
- 테이블에서는 `이름/나이·성별`, `이메일/휴대폰`, `주소/국가`, `회원유형/구독`처럼 2줄 묶음 컬럼을 유지합니다.
- Drawer 상세에서는 편집성을 위해 묶음 컬럼을 다시 `이름`, `나이`, `성별`, `이메일`, `휴대폰`, `주소`, `국가`, `회원유형`, `구독상태`처럼 나눕니다.
- Drawer는 닫기 버튼, ESC 닫기, 선택 row 해제, focus 복귀를 포함합니다.
- 이 패턴의 공통 구현은 `table_design/table-design.css`, `table_design/table-design.js`에 둡니다.

## 5. 레이아웃 패턴

### 5.1 기본 페이지

페이지 레이아웃 섹션처럼 콘텐츠 영역 안에서 한 번 더 구분이 필요하면 하위 탭을 사용합니다.

- 상위 콘텐츠 탭은 `페이지 레이아웃`, `MD 문서 보기`, `신청서폼`처럼 큰 화면 단위를 나눕니다.
- 하위 탭은 선택된 콘텐츠 안에서 `기본 구조`, `BO 목록 화면`, `인증·폼 화면`처럼 레이아웃 유형을 나눕니다.
- 하위 탭은 상단 세부 목차 바로 아래에 `section-subtabs`로 노출합니다.
- 특정 상위 탭에만 하위 탭이 필요할 때는 `data-section-subtabs="상위섹션ID"`와 anchor 링크를 연결합니다.
- 하위 탭은 3depth 이후 콘텐츠 탐색에 해당하므로 사이드바나 상위 탭을 늘리지 않고 본문 안에서 처리합니다.
- 현재 `index.html`의 좌측 사이드바에는 `BO/DB` 분류가 있으며, 관리자 테이블 실물 페이지와 DB v2 원본 참고 화면을 iframe으로 확인합니다.

- `shell`: 전체 문서 구조
- `nav`: 디자인 시스템 문서 목차
- `main`: 본문
- `section`: 문서 또는 화면 단위
- `panel`: 경계가 있는 기본 패널
- `grid two`, `grid three`, `grid four`: 반응형 그리드

### 5.2 BO/DB 페이지 허브

`index.html`의 `BO/DB` 섹션은 디자인 시스템 안에서 관리자 테이블 페이지를 바로 확인하는 링크 허브입니다.

- `table_design/관리자패널_전체회원.html`: 50명 회원 데이터, DB v2 검색조건, inline 우측 Dense Drawer
- `table_design/신규테이블관리.html`: 여러 신규 테이블을 전환하는 BO 테이블 페이지
- `table_design/결제관리.html`: 결제/환불/정산 테이블 페이지
- `resource/DB_v2_HTML/DB페이지_v2(이거사용).html`: DB v2 원본 참고 화면
- `resource/DB_v2_HTML/관리자패널_전체회원.html`, `resource/DB_v2_HTML/결제관리.html`: 원본 비교 화면

새 BO 페이지를 추가하면 `index.html`의 `BO/DB 페이지 보기` 목록에 `data-page-src`, `data-page-title` 버튼을 추가합니다. 미리보기 전환은 `common.js`의 `[data-page-src]` 핸들러가 담당합니다.

### 5.3 서비스 홈페이지

홈페이지는 과한 장식보다 실제 사용자가 바로 이해할 수 있는 정보 구조를 우선합니다.

권장:

- Hero는 서비스명/서비스 소개를 첫 신호로 보여줍니다.
- CTA는 "전문가 신청", "견적 의뢰", "상담하기"처럼 실제 행동 문구로 씁니다.
- 카드가 필요한 경우 반복 항목에만 사용합니다.
- 검색/필터가 있는 화면은 인라인 칩과 활성 태그를 함께 제공합니다.

지양:

- 회사 소개를 첫 화면의 핵심으로 두는 구성
- 상태가 색상만으로 구분되는 칩/배지
- 좁은 입력칸과 한 화면 과밀 배치
- 동일 역할의 버튼 class를 페이지마다 새로 만드는 방식

## 6. 새 홈페이지 제작 체크리스트

- 공통 `layout.css` 토큰을 먼저 연결했는가?
- Header, Content, Footer 구조가 유지되는가?
- 서비스 소개가 회사 소개보다 먼저 나오는가?
- 실적, 전문가 신청, 견적 의뢰가 빠지지 않았는가?
- 버튼은 `btn btn--variant btn--size` 조합을 쓰는가?
- 칩/상태/badge/tag의 대비가 최소 `3:1` 이상인가?
- 선택 상태에 `aria-pressed` 또는 `aria-selected`가 있는가?
- 키보드 포커스가 보이는가?
- 모바일에서 테이블/필터/헤더가 깨지지 않는가?
- 색상 직접 입력보다 토큰 사용을 우선했는가?
