# 사이드바 기준 프롬프트

## 목적

BO(Back Office) 앱 전체에서 재사용 가능한 좌측 사이드바 컴포넌트의 기준을 정의한다.  
이 문서는 사이드바 구현 전에 정보 구조, 시각 규칙, 상태 표현, 반응형 동작을 맞추기 위한 기준 프롬프트로 사용한다.

## 기준 프롬프트

```text
BO용 사이드바 UI 기준을 설계한다.

목표:
- BO 앱 전체에서 재사용 가능한 좌측 사이드바 컴포넌트를 만든다.
- 정보 구조, 시각 규칙, 상태 표현, 반응형 동작 기준을 명확히 정의한다.
- BO 화면 특성에 맞게 장식은 줄이고, 정보 밀도와 반복 사용성을 우선한다.

포함할 항목:
1. 사이드바의 역할
   - 전역 내비게이션
   - 현재 위치 표시
   - 주요 기능 접근
   - 1depth와 2depth까지만 탐색 제공
   - 3depth 이후 하위 분류는 콘텐츠 영역에서 제공
   - 보조 액션 또는 계정 영역 포함 여부

2. 레이아웃 기준
   - 기본 너비
   - 접힘 상태 너비
   - 헤더, 메뉴, 하단 영역 구분
   - 아이콘과 텍스트 정렬
   - 메뉴 간격과 패딩은 최소화
   - BO 화면 기준으로 한 화면에 더 많은 메뉴와 콘텐츠가 보이도록 설계

3. 메뉴 구조
   - 사이드바에는 대분류인 1depth와 중분류인 2depth까지만 표시
   - 1depth 메뉴 기준
   - 2depth 메뉴 표시 방식
   - 3depth 이후 분류는 사이드바에 노출하지 않고 콘텐츠 본문 안에서 탭, 리스트, 필터, 섹션 내비게이션 등으로 처리
   - 그룹 라벨 사용 여부
   - 메뉴 개수가 많아질 때의 스크롤 처리

4. 상태 정의
   - 기본 상태
   - hover
   - active/current
   - disabled
   - focus
   - collapsed
   - expanded

5. 디자인 토큰
   - 배경색
   - 텍스트 색
   - 아이콘 색
   - active 배경
   - border 또는 divider
   - border-radius는 0 사용
   - spacing은 조밀하게 사용
   - font size / weight

6. 접근성
   - 키보드 탐색 가능
   - focus ring 표시
   - 현재 페이지 aria-current 적용
   - 아이콘만 보이는 경우 aria-label 또는 tooltip 제공

7. 반응형 기준
   - desktop: 고정 사이드바
   - tablet: 접힘 또는 오버레이
   - mobile: drawer 방식
   - breakpoint 기준

8. 구현 산출물
   - 사이드바 컴포넌트
   - 메뉴 데이터 구조
   - active route 처리 방식
   - collapsed 상태 관리
   - 콘텐츠 영역의 3depth 이하 분류 표시 패턴
   - 예시 메뉴 포함
```

## BO 시각 기준

- BO 버전은 관리자가 반복적으로 사용하는 업무 화면을 기준으로 한다.
- 장식적인 카드, 과한 여백, 큰 타이포그래피를 피하고 정보 탐색과 입력 효율을 우선한다.
- `border-radius: 0`을 기본값으로 한다.
- 사이드바, 버튼, 탭, 패널, 목록, 상태 배지 모두 둥근 모서리를 사용하지 않는다.
- 간격은 최소화한다. 메뉴와 목록은 높이를 낮추고, 본문 패딩도 과하게 넓히지 않는다.
- 색상은 흰색/회색 계열의 배경, 명확한 border, 제한적인 brand 색상으로 구성한다.
- active 상태만 명확히 강조하고 hover는 약하게 표현한다.
- BO 화면에서는 콘텐츠를 카드처럼 띄우기보다 표, 목록, 구획선 중심으로 정리한다.

## 정보 구조 기준

- 사이드바는 전체 서비스의 큰 구조를 잡는 영역으로 사용한다.
- 사이드바에는 1depth 대분류와 2depth 중분류까지만 표시한다.
- 3depth부터 그 이하 하위 분류는 콘텐츠 영역 안에서 표시한다.
- 2depth를 선택하면 콘텐츠 영역 상단 또는 본문 안에서 3depth 이하 항목을 탐색할 수 있어야 한다.
- 현재 위치 표시는 사이드바의 1depth/2depth active 상태와 콘텐츠 영역의 3depth 이하 active 상태가 함께 맞물리도록 한다.
- 3depth 이하를 사이드바에 중첩 표시하지 않는다. 사이드바가 깊어져서 탐색 구조가 복잡해지는 것을 방지한다.

## BO 대분류/중분류 정리 프롬프트

아래 프롬프트는 HutechC 사내 Studio 개선 문서의 사이드바 단순화 원칙을 BO 기준으로 재사용하기 위한 기준이다.

```text
BO 사이드바의 대분류와 중분류를 정리한다.

참고 기준:
- 현재처럼 여러 업무 메뉴를 같은 층위에 평면 나열하지 않는다.
- 최상위 대분류는 5개 안팎으로 제한한다.
- 즐겨찾기, 시스템 관리, 보관함은 일반 업무 대분류와 분리한다.
- 같은 데이터의 여러 보기/관리/편집/레이아웃 화면은 사이드바 메뉴로 쪼개지 않고 콘텐츠 영역의 탭이나 보기 전환으로 통합한다.
- 권한이 높은 메뉴와 민감한 인사/채용 메뉴는 시각적으로도 분리한다.
- 이전 메뉴와 대체 메뉴는 현역 메뉴와 섞지 않고 보관함(Legacy)으로 격리한다.

추천 대분류:
1. 홈 · 내 업무
   - 대시보드
   - 업무일지
   - 업무 총괄
   - 내 출퇴근

2. 지식 · 규정
   - 사내업무지침
   - 규정
   - 매뉴얼
   - 서식 · 확인서
   - 업무 자료

3. 인사 · 채용
   - 채용관리
   - 출퇴근 관리
   - 학생관리
   - 출강 · 강사관리

4. 운영 · 거래처
   - 미팅 · 일정
   - 강의시간표
   - 거래처관리
   - 미수금관리
   - 프로젝트관리
   - 사진모음

5. 기획 · 콘텐츠
   - 업무 분류
   - 강사 커리큘럼
   - 마케팅
   - 프롬프트 사용법
   - AI 강의안
   - 커뮤니티 · 홈페이지

부가 영역:
- 즐겨찾기: 사용자별 자주 쓰는 5~7개 메뉴를 상단에 고정한다.
- 시스템 관리: 관리자 전용 메뉴를 일반 업무와 분리한다.
- 보관함(Legacy): 이전/대체 메뉴를 기본 접힘 상태로 격리한다.

콘텐츠 영역 처리:
- 3depth 이후는 사이드바에 넣지 않는다.
- 예: 규정 메뉴 안에서 통합/관리/편집/레이아웃/매뉴얼/평가기준은 콘텐츠 탭으로 처리한다.
- 예: 거래처관리 안에서 거래처 목록/아웃콜/출장 거래처/계약 이력은 콘텐츠 탭으로 처리한다.
- 예: 채용관리 안에서 지원서/면접/대시보드/평가는 콘텐츠 탭으로 처리한다.
```

## BO 분류 정리 이유

- 사용자는 먼저 업무 도메인을 선택하고, 그 다음 세부 화면으로 들어가는 방식이 더 빠르다.
- 49개 안팎의 메뉴를 한 층에 펼치면 검색보다 스크롤과 기억에 의존하게 된다.
- 규정, 채용, 거래처처럼 같은 데이터를 다루는 화면은 사이드바 메뉴가 아니라 한 화면 안의 탭으로 묶어야 데이터의 소유권이 분명해진다.
- 시스템 관리와 인사/채용은 권한·민감도 기준이 다르므로 일반 업무와 시각적으로 분리하는 것이 안전하다.
- 보관함을 두면 이전 화면을 완전히 삭제하지 않고도 현역 업무 흐름에서 분리할 수 있다.
- BO 화면은 반복 사용성이 중요하므로 대분류는 적게, 중분류는 명확하게, 3depth 이후는 본문에서 처리하는 편이 유지보수에 유리하다.

### 구조 예시

```text
Sidebar
- 1depth: 시험 운영
  - 2depth: 기본 설정
  - 2depth: 응시 대상
  - 2depth: 출제/채점

Content
- 3depth: 시험 문항 구성
- 3depth: 채점 방식
  - 4depth: 자동 채점
  - 4depth: 수동 채점
- 3depth: 일정 운영
```

## 구현용 축약 프롬프트

```text
BO용 좌측 사이드바 컴포넌트를 만든다.
구성은 상단 로고/타이틀, 중앙 내비게이션 메뉴, 하단 사용자/설정 영역으로 나눈다.
사이드바 메뉴는 1depth 대분류와 2depth 중분류까지만 표시한다.
3depth부터 그 이하 하위 분류는 사이드바에 넣지 않고 콘텐츠 영역 안에서 탭, 리스트, 필터, 섹션 내비게이션 등으로 표시한다.
사이드바 메뉴는 아이콘 + 라벨 구조이며 active, hover, disabled, focus 상태를 가진다.
desktop에서는 260px 안팎의 고정 너비, collapsed 상태에서는 56px 안팎의 너비를 사용한다.
mobile에서는 오버레이 drawer로 동작한다.
현재 경로와 일치하는 메뉴에는 aria-current="page"를 적용한다.
현재 위치는 사이드바의 1depth/2depth active 상태와 콘텐츠 영역의 3depth 이하 active 상태가 함께 드러나도록 한다.
BO 화면 기준으로 border-radius는 0을 사용하고, 메뉴 높이와 패딩은 최소화한다.
디자인은 기존 프로젝트의 토큰, 색상, 간격, typography 규칙을 우선 사용하되 radius는 0으로 통일한다.
장식적인 카드형 UI보다 border, table, list, tab 중심의 조밀한 레이아웃을 사용한다.
```

## 재사용 프롬프트

```text
BO 관리 화면용 사이드바 샘플을 만든다.

정보 구조:
- 사이드바는 1depth 대분류와 2depth 중분류까지만 표시한다.
- 3depth부터 하위 분류는 콘텐츠 영역에서 탭, 필터, 리스트, 표, 섹션 내비게이션 등으로 표시한다.
- 2depth를 선택하면 콘텐츠 영역의 3depth 탐색과 본문 내용이 갱신된다.
- 현재 위치는 breadcrumb, 사이드바 active, 콘텐츠 3depth active가 함께 맞아야 한다.

시각 기준:
- BO 화면이므로 정보 밀도를 높인다.
- border-radius는 모든 UI에서 0으로 통일한다.
- 여백, 메뉴 높이, 패딩, 카드 간격을 최소화한다.
- 장식적인 그림자와 큰 카드형 표현은 피한다.
- 흰색/회색 배경, 얇은 border, 제한적인 brand color만 사용한다.
- active 상태는 명확히, hover 상태는 약하게 표현한다.

구현 기준:
- 단일 HTML 샘플로 만든다.
- 사이드바, 3depth 탭, 4depth 이하 목록/표가 한 화면에서 구조적으로 보이게 한다.
- mobile에서는 사이드바를 drawer로 전환한다.
- aria-current, aria-label 등 기본 접근성 속성을 포함한다.
```

## 검토 포인트

- 사이드바가 필요한 화면 범위
- 사이드바는 2depth까지만 허용하고 3depth 이후는 콘텐츠 영역에서 처리하는 기준
- 콘텐츠 영역에서 3depth 이하를 탭, 필터, 리스트, 섹션 내비게이션 중 어떤 패턴으로 보여줄지
- collapsed 상태가 실제로 필요한지 여부
- 하단 영역에 계정, 설정, 도움말 중 무엇을 둘지
- mobile에서 drawer를 사용할지, 하단 내비게이션으로 바꿀지
- 기존 디자인 시스템 토큰과 충돌하는 수치가 있는지
- BO 버전에서 모든 컴포넌트의 border-radius가 0으로 유지되는지
- BO 버전에서 여백이 과하게 넓어지지 않고 정보 밀도가 유지되는지
