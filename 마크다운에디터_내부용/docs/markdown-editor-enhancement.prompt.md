# 마크다운 에디터 고도화 프롬프트

## 작업 프롬프트

브라우저에서 바로 실행되는 마크다운 에디터와 Word 변환기를 고도화한다.

현재 앱 상태:
- 단일 파일 앱: `markdown-to-word.html`
- HTML, CSS, JavaScript만 사용
- 빌드 과정 없음
- 백엔드 없음
- 외부 패키지 의존성 없음
- 마크다운 파일 드래그 앤 드롭 지원
- 여러 파일 선택 지원
- 변환 전 미리보기 지원
- 마크다운 블록을 `.docx` 파일로 변환
- 여러 문서를 변환할 때 ZIP 다운로드 지원

핵심 목표:
- 단순한 Markdown to Word 변환기를 실무용 마크다운 에디터로 확장한다.
- 작성, 미리보기, Word 내보내기, 문서 형식 조정이 한 화면에서 자연스럽게 이어지도록 만든다.

작업 원칙:
- 특별한 이유가 생기기 전까지 브라우저 전용 앱 구조를 유지한다.
- 기존 Word 변환 기능은 깨지지 않게 보존한다.
- 기능은 한 번에 크게 갈아엎지 말고 검증 가능한 단위로 추가한다.
- UI 문구는 한국어로 짧고 명확하게 작성한다.
- 장식적인 화면보다 문서 작성과 검토 효율을 우선한다.
- 에디터 흐름과 내보내기 요구사항이 확정되기 전에는 대규모 구조 변경을 피한다.

우선 고도화 방향:
- 파일 업로드뿐 아니라 직접 마크다운을 작성하고 수정할 수 있게 한다.
- 입력한 내용과 미리보기가 실시간으로 동기화되게 한다.
- 현재 편집 중인 문서를 바로 Word로 내보낼 수 있게 한다.
- 제목, 저장 파일명, 글꼴, 여백, 표 스타일 같은 문서 설정을 추가한다.
- 브라우저 로컬 임시 저장 기능을 추가한다.
- 모바일에서도 파일 선택, 편집, 미리보기, 내보내기 흐름이 무너지지 않게 한다.

## 댓글 고도화 프롬프트

마크다운 편집창에서 드래그로 선택한 영역에 댓글이 달리면, 미리보기에서도 해당 텍스트가 파란톤으로 표시되어야 한다.

댓글 표시 원칙:
- 댓글이 달린 텍스트는 미리보기에서 파란톤 하이라이트로 표시한다.
- 하이라이트에는 댓글 작성자 수 또는 댓글 개수를 알 수 있는 작은 표시를 둔다.
- 같은 영역에 여러 명이 댓글을 달 수 있다.
- 여러 명이 댓글을 단 경우, 해당 영역에 연결된 댓글을 모두 목록으로 볼 수 있어야 한다.
- 댓글 목록에는 작성자, 작성내용, 작성 시각, 연결된 선택 문구가 보여야 한다.
- 댓글이 달린 미리보기 영역을 누르면 오른쪽 댓글 패널에서 해당 댓글 묶음이 강조되어야 한다.
- 댓글 데이터는 작성자, 작성내용, 선택 텍스트, 선택 시작 위치, 선택 종료 위치를 유지한다.
- 문서 내용이 수정되어 위치가 어긋날 수 있으므로, 우선 선택 텍스트 기반으로도 하이라이트를 복구한다.

이번 구현 범위:
- 기존 댓글 저장 API는 유지한다.
- 미리보기 렌더링 시 댓글 선택 영역을 하이라이트한다.
- 오른쪽 댓글 패널을 선택 영역 기준으로 묶어서 보여준다.
- 같은 선택 영역에 여러 댓글이 있으면 하나의 묶음 안에 모두 표시한다.
- 하이라이트 클릭 시 해당 댓글 묶음으로 스크롤하고 강조한다.

다음 개선 후보:
- 댓글 수정 및 삭제
- 댓글 해결됨 상태
- 하이라이트 색상 구분
- 문서 수정 후 댓글 위치 재매핑 정확도 개선
- Word 내보내기에 댓글 정보를 주석 또는 별도 표로 포함

## 대댓글 및 파일첨부 고도화 프롬프트

댓글에는 대댓글과 파일첨부가 가능해야 한다.

대댓글 원칙:
- 사용자는 기존 댓글에 답글을 달 수 있어야 한다.
- 대댓글에는 작성자, 작성내용, 작성 시각이 표시되어야 한다.
- 대댓글은 원 댓글 아래에 계층적으로 표시한다.
- 대댓글은 원 댓글이 연결된 선택 영역과 같은 댓글 묶음 안에서 보여준다.
- 대댓글이 여러 개일 경우 작성 시각 순서로 보여준다.

파일첨부 원칙:
- 댓글과 대댓글 모두 파일을 첨부할 수 있는 구조로 만든다.
- 첨부 파일은 서버의 `uploads/attachments/` 폴더에 저장한다.
- DB에는 원본 파일명, 저장 파일명, 파일 크기, MIME 타입, 연결된 댓글 ID를 저장한다.
- 댓글 목록에는 첨부 파일명을 표시한다.
- 우선 다운로드 링크 제공은 보류하고, 파일 저장과 목록 표시까지 구현한다.

이번 구현 범위:
- 댓글 DB 구조에 `parentId`와 첨부파일 목록을 추가한다.
- 기존 원댓글은 `parentId: null`로 저장한다.
- 대댓글은 같은 API에서 `parentId`를 받아 저장한다.
- 댓글 작성 폼과 대댓글 작성 폼에서 첨부 파일을 함께 보낼 수 있게 한다.
- 기존 댓글 하이라이트와 댓글 묶음 기능은 유지한다.

## 댓글 하이라이트 UX 및 채팅 고도화 프롬프트

작성 영역에서 드래그한 원문 위치가 미리보기에서 명확히 보여야 한다.

하이라이트 UX 원칙:
- 댓글을 달 때 사용자가 드래그한 정확한 원문 범위를 저장한다.
- 댓글 작성자가 작성 탭에서 텍스트를 드래그하는 즉시, 같은 영역이 미리보기에도 표시되어야 한다.
- 저장 전 선택 영역은 댓글 작성 중임을 알 수 있도록 댓글 하이라이트보다 더 선명하게 표시한다.
- 미리보기는 저장된 시작/끝 위치를 기준으로 댓글 영역을 파란톤으로 표시한다.
- 마크다운 문법 때문에 미리보기 텍스트가 달라져도 가능한 한 같은 위치가 표시되어야 한다.
- 하이라이트에는 댓글 개수를 표시한다.
- 하이라이트를 클릭하면 오른쪽 댓글 묶음이 강조된다.
- 댓글 묶음을 클릭하면 연결된 미리보기 하이라이트로 스크롤하고 강조된다.
- 댓글 하나만 있을 때도 어느 영역의 댓글인지 즉시 알 수 있어야 한다.
- 문서가 수정되어 위치가 어긋난 경우에는 선택 텍스트 기반 복구를 시도한다.

채팅 원칙:
- 문서별 채팅 공간을 제공한다.
- 채팅에는 작성자, 메시지, 작성 시각이 저장된다.
- 채팅은 댓글과 별개로 문서 전체에 대한 대화로 사용한다.
- 채팅 패널은 댓글 패널과 같은 오른쪽 영역 안에서 접근 가능해야 한다.
- 우선 실시간 웹소켓은 사용하지 않고 저장/조회형 채팅으로 구현한다.

이번 구현 범위:
- 원본 마크다운 위치 기반 미리보기 하이라이트를 구현한다.
- 작성 탭에서 드래그한 현재 선택 영역을 미리보기에서 즉시 하이라이트한다.
- 댓글 관련 하이라이트 색상은 파란톤으로 통일한다.
- 댓글 패널에서 댓글 묶음을 누르면 연결된 미리보기 하이라이트로 이동한다.
- 하이라이트와 댓글 묶음의 양방향 포커스를 구현한다.
- 문서별 채팅 저장/조회 API를 추가한다.
- 오른쪽 패널에 댓글/채팅 탭을 추가한다.

## 열린 문서 탭 고도화 프롬프트

사용자는 여러 마크다운 문서 중 일부만 작업 중일 수 있으므로, IDE처럼 열린 문서를 상단 탭으로 빠르게 전환할 수 있어야 한다.

열린 탭 UX 원칙:
- 왼쪽 문서 목록은 전체 문서 탐색 역할을 유지한다.
- 사용자가 문서를 열면 상단 열린 문서 탭에 추가한다.
- 이미 열린 문서를 다시 열면 새 탭을 만들지 않고 기존 탭을 활성화한다.
- 상단 탭에는 문서 제목 또는 파일명이 표시된다.
- 현재 활성 문서 탭은 명확히 강조한다.
- 탭에는 닫기 버튼이 있어야 한다.
- 활성 탭을 닫으면 인접한 열린 탭으로 이동한다.
- 마지막 탭을 닫으면 새 문서 작성 상태로 전환한다.
- 모바일에서는 탭이 가로 스크롤되어야 한다.

이번 구현 범위:
- 에디터 상단에 열린 문서 탭 영역을 추가한다.
- 문서 열기, 새 문서 생성, 업로드 후 열기 시 탭 상태를 갱신한다.
- 열린 탭 클릭으로 문서를 빠르게 전환한다.
- 탭 닫기 기능을 추가한다.

## 내부용 로그인 및 삭제 권한 프롬프트

내부에서만 쓰는 앱이므로 복잡한 인증보다는 간단하고 안전한 로컬 인증을 제공한다.

인증 원칙:
- 회원가입과 로그인을 제공한다.
- 사용자는 사용자명과 비밀번호로 가입한다.
- 비밀번호는 평문 저장하지 않고 해시로 저장한다.
- 로그인 후 세션 쿠키로 현재 사용자를 식별한다.
- 화면 상단에 현재 로그인 사용자를 표시한다.
- 로그아웃을 제공한다.

삭제 권한 원칙:
- 사용자가 직접 작성한 문서, 댓글, 대댓글, 채팅은 삭제할 수 있어야 한다.
- 내부용이므로 우선 작성자 본인만 삭제 가능하게 한다.
- 기존 작성자 정보가 없는 데이터는 삭제 버튼을 표시하지 않는다.
- 문서를 삭제하면 해당 문서의 댓글, 대댓글, 채팅도 함께 정리한다.

열린 탭 보정 원칙:
- 상단 열린 문서 탭 영역은 항상 보여야 한다.
- 열린 탭이 없을 때도 안내 문구를 표시한다.
- 문서를 여러 개 열면 즉시 상단 탭에 표시되어야 한다.
- 문서 삭제 시 해당 문서 탭도 닫혀야 한다.

다음 대화 흐름:
1. 첫 화면을 어떤 에디터 구조로 만들지 정한다.
2. 파일 업로드와 직접 편집을 한 화면에 둘지, 탭으로 나눌지 정한다.
3. 최소 기능부터 추가한다: 입력창, 실시간 미리보기, 임시 저장, 현재 문서 Word 내보내기.
4. 기존 `.docx` 변환 기능이 그대로 동작하는지 확인한다.
5. Word 문서 품질을 제목, 목록, 표, 코드블록, 인용문, 링크 순서로 다듬는다.

결정할 사항:
- 앱 이름을 `Markdown to Word 변환기`로 유지할지, `마크다운 에디터`로 바꿀지
- Word 결과물을 업무 보고서 스타일로 만들지, 읽기 좋은 기본 문서 스타일로 만들지
- 여러 파일 일괄 변환을 핵심 기능으로 둘지, 보조 기능으로 분리할지
