# Figma MCP 사용 가이드 및 통합 화면 구현 전략

작성일: 2026-05-21  
대상 프로젝트: 시험 플랫폼 Figma/PDF 분석  
기준 사례: Figma node `5:27589` / `목록_시험접수`

## 1. 먼저 결론

Figma MCP는 “Figma 파일 전체를 한 번에 완벽하게 읽어서 전체 사이트를 자동 구현하는 도구”라기보다, **선택한 화면 또는 선택한 노드의 구조와 시각 정보를 정확히 가져오는 도구**에 가깝다.

따라서 가장 좋은 방식은 다음 순서다.

1. 전체 파일 또는 페이지 구조를 먼저 파악한다.
2. 대표 화면을 하나씩 선택해서 정확한 디자인 기준을 추출한다.
3. 반복되는 공통 구조를 컴포넌트로 정리한다.
4. 화면별 차이만 따로 정리한다.
5. 마지막에 사이트맵, 기능 정의, UI 컴포넌트, 화면 구현 우선순위를 통합한다.

즉, 화면을 하나씩 선택하는 과정은 번거로운 일이 아니라 **정확한 구현을 위한 기준 화면을 수집하는 과정**으로 보는 것이 좋다.

## 2. Figma MCP로 무엇을 할 수 있는가?

Figma MCP는 Figma 화면을 분석하거나 구현에 참고할 정보를 가져올 때 사용한다.

| 할 수 있는 일 | 설명 | 사용 상황 |
|---|---|---|
| 스크린샷 추출 | 선택한 프레임/노드의 실제 시각 모습을 이미지처럼 확인 | 디자인이 실제로 어떻게 생겼는지 확인할 때 |
| 디자인 컨텍스트 추출 | 레이아웃, 텍스트, 색상, 크기, 계층 구조를 코드 형태로 참고 | HTML/CSS 또는 React 구현 기준을 잡을 때 |
| 메타데이터 확인 | 페이지 안의 노드 목록, 이름, 크기, 위치를 확인 | 어떤 화면들이 들어 있는지 구조를 볼 때 |
| 디자인 시스템 검색 | 버튼, 입력창, 테이블 등 기존 컴포넌트가 있는지 확인 | 같은 컴포넌트를 재사용해야 할 때 |
| Figma 파일에 쓰기 | 새 프레임, 컴포넌트, 토큰 등을 직접 생성/수정 | Figma 안에 새 디자인을 만들거나 고칠 때 |

이번 프로젝트에서는 주로 **스크린샷 추출**과 **디자인 컨텍스트 추출**이 중요하다. 이유는 PDF만 볼 때보다 실제 Figma 노드의 레이아웃 계층과 화면 비율을 더 정확히 알 수 있기 때문이다.

## 3. 페이지를 통째로 선택하면 안 되는가?

가능은 하지만, 항상 좋은 방식은 아니다.

Figma에서 페이지 전체 또는 너무 큰 그룹을 선택하면 다음 문제가 생길 수 있다.

| 문제 | 설명 |
|---|---|
| 정보가 너무 많아짐 | 수십 개 화면이 한 번에 들어오면 핵심 구조를 구분하기 어렵다 |
| 일부 정보가 생략될 수 있음 | MCP 응답 크기가 커져서 코드나 텍스트가 일부만 반환될 수 있다 |
| 화면 단위가 흐려짐 | 로그인, 목록, 상세, 팝업, 관리자 화면이 섞여 해석될 수 있다 |
| 구현 기준이 흔들림 | 어떤 화면을 기준 디자인으로 삼아야 하는지 불명확해진다 |

그래서 페이지 전체 선택은 **탐색용**으로는 괜찮지만, 실제 구현 기준을 잡을 때는 화면 단위로 선택하는 것이 좋다.

## 4. 하나씩 선택하고 호출해야 하는가?

정확한 답은 “중요 화면은 하나씩 선택하는 것이 좋고, 모든 화면을 하나씩 깊게 분석할 필요는 없다”이다.

화면을 세 종류로 나누면 된다.

| 화면 유형 | MCP 호출 방식 | 예시 |
|---|---|---|
| 기준 화면 | 반드시 개별 선택 후 상세 추출 | `목록_시험접수`, 관리자 홈, 시험응시 화면 |
| 변형 화면 | 기준 화면과 비교해 차이만 확인 | 접수 변경, 접수 취소, 상세 팝업 |
| 단순 화면 | PDF/파일명 분석으로 충분할 수 있음 | 이용약관, 저장 완료, 정보 없음 |

이번에 확인한 `목록_시험접수`는 기준 화면이다. 이 화면에는 2024 디자인의 핵심 구조가 들어 있다.

- 상단 글로벌 메뉴
- 좌측 아이콘 레일
- 마이페이지 사이드바
- 사용자 프로필
- 검색 조건
- 시험 정보 테이블
- 상태별 버튼
- 페이지네이션
- 푸터

이런 화면은 반드시 MCP로 직접 확인해야 한다. 반대로 `저장팝업`, `로그아웃`, `정보없을경우` 같은 단순 화면은 기준 화면을 먼저 잡은 뒤 필요한 경우에만 확인해도 된다.

## 5. 권장 MCP 분석 순서

시험 플랫폼처럼 화면이 많은 프로젝트에서는 다음 순서가 좋다.

### 5.1 1단계: 전체 구조 파악

먼저 PDF 폴더, 파일명, 메뉴 구조 이미지를 기준으로 전체 화면 목록을 만든다.

이 단계의 목표는 디자인을 구현하는 것이 아니라, “어떤 화면이 존재하는지”를 정리하는 것이다.

확인할 것:

- 사용자 역할: 수험생, 출제자, 채점자, 관리자
- 업무 흐름: 접수, 결제, 응시, 출제, 채점, 결과, 자격증
- 화면 유형: 목록, 상세, 등록, 수정, 팝업, 완료, 빈 상태
- 반복 패턴: 테이블, 검색, 사이드 메뉴, 버튼, 상태값

### 5.2 2단계: 기준 화면 선정

모든 화면을 같은 깊이로 분석하면 시간이 많이 들고, 오히려 판단이 흐려진다. 먼저 기준 화면을 고른다.

| 영역 | 우선 확인할 기준 화면 |
|---|---|
| 수험생 | 시험접수 목록, 시험응시, 시험결과, 자격증 신청 |
| 출제자 | 출제관리 목록, 번역/단답형/프롬프트 출제 상세 |
| 채점자 | 채점관리 목록, 번역/단답형/프롬프트 채점 상세 |
| 관리자 | 관리자 홈, 회원관리, 시험관리, 결제관리, 문의관리 |

이 기준 화면들은 MCP로 직접 선택해서 스크린샷과 디자인 컨텍스트를 추출하는 것이 좋다.

### 5.3 3단계: 공통 레이아웃 추출

기준 화면을 몇 개 보면 반복되는 구조가 보인다.

2024 기준으로는 다음 공통 레이아웃이 중요하다.

| 공통 구조 | 설명 |
|---|---|
| 상단 헤더 | 로고, 글로벌 메뉴, 사용자 도구 |
| 좌측 레일 | 빠른 접근 아이콘 또는 보조 메뉴 |
| 마이페이지/관리자 사이드바 | 역할별 하위 메뉴 |
| 검색 조건 영역 | 날짜, 검색어, 상태, 버튼 |
| 데이터 테이블 | 목록 화면의 핵심 정보 표시 |
| 액션 버튼 | 접수, 변경, 취소, 저장, 상세, 다운로드 |
| 페이지네이션 | 목록 화면 하단 이동 |
| 푸터 | 회사 정보, 정책 링크 |

이 공통 구조를 먼저 구현하면, 모든 화면을 새로 만드는 것이 아니라 “같은 틀 안에서 내용만 바꾸는 방식”으로 구현할 수 있다.

### 5.4 4단계: 화면별 차이 정리

기준 레이아웃을 만든 뒤에는 화면별 차이만 정리한다.

예를 들어 시험접수 목록과 시험결과 목록은 둘 다 목록 화면이지만, 테이블 컬럼과 버튼이 다르다.

| 화면 | 공통점 | 차이점 |
|---|---|---|
| 시험접수 | 검색 조건, 테이블, 페이지네이션 | 접수하기, 변경, 취소 버튼 |
| 시험응시 | 검색 조건, 테이블, 페이지네이션 | 시험 시작, 이어보기, 제출 상태 |
| 시험결과 | 검색 조건, 테이블, 페이지네이션 | 점수, 합격 여부, 결과 상세 |
| 자격증신청 | 검색 조건, 테이블, 페이지네이션 | 자격증 신청, 발급 상태, 결제 여부 |

이렇게 정리하면 전체 구현량이 크게 줄어든다.

## 6. 통합적인 모든 화면 구현은 어떻게 처리하는 것이 좋은가?

전체 화면 구현은 “Figma 화면을 1:1로 무작정 많이 만드는 방식”보다 **구조화된 화면 시스템**으로 접근하는 것이 좋다.

권장 방식은 다음과 같다.

### 6.1 화면을 먼저 그룹화한다

시험 플랫폼 화면은 다음 그룹으로 나눌 수 있다.

| 그룹 | 포함 화면 |
|---|---|
| 인증/계정 | 로그인, 회원가입, 아이디 찾기, 비밀번호 재설정, 본인인증 |
| 수험생 마이페이지 | 시험접수, 시험응시, 시험결과, 자격증신청, 결제내역, 문의 |
| 시험 응시 | 시험 시작, 시험중, 제출, 종료, 단답형, 번역형, 프롬프트형 |
| 출제자 | 출제관리, 출제 상세, 임시저장, 출제내역 |
| 채점자 | 채점관리, 채점 상세, 피드백, 채점내역 |
| 관리자 | 관리자 홈, 회원관리, 시험관리, 문제관리, 채점관리, 결제관리, 정산관리, 통계 |
| 운영/콘텐츠 | 게시판, FAQ, 블로그, 배너, 알림, 이메일 |

이 그룹 기준으로 구현하면 화면 수가 많아도 관리가 가능하다.

### 6.2 화면 유형별 템플릿을 만든다

개별 화면을 모두 따로 구현하는 대신, 화면 유형별 템플릿을 만든다.

| 템플릿 | 쓰이는 화면 |
|---|---|
| 로그인/인증 템플릿 | 로그인, 회원가입, 비밀번호 찾기 |
| 마이페이지 목록 템플릿 | 시험접수, 시험응시, 시험결과, 결제내역 |
| 관리자 목록 템플릿 | 회원관리, 시험관리, 결제관리, 문의관리 |
| 상세/수정 템플릿 | 회원 상세, 시험 상세, 전문가 상세 |
| 작업형 템플릿 | 시험 응시, 출제, 채점, 프롬프트 작업 |
| 팝업 템플릿 | 접수 확인, 저장 확인, 제출 확인, 문의 팝업 |

Figma MCP는 이 템플릿의 기준이 되는 대표 화면을 정확히 추출하는 데 사용한다.

### 6.3 모든 화면을 MCP로 깊게 추출하지 않는다

모든 화면을 MCP로 하나씩 깊게 추출하면 시간이 오래 걸리고, 구현 관점에서는 중복이 많다.

대신 다음 기준으로 나눈다.

| 우선순위 | 처리 방식 |
|---|---|
| 높음 | MCP 상세 추출 후 구현 기준으로 사용 |
| 중간 | 스크린샷만 확인하고 기준 템플릿에 맞춰 변형 |
| 낮음 | PDF/파일명/문서 분석으로 기능만 정의 |

예를 들어 `목록_시험접수`를 기준으로 마이페이지 목록 템플릿을 만든 뒤, `목록_시험응시`, `목록_시험결과`, `목록_자격증신청`은 컬럼과 버튼만 바꾸는 방식이 효율적이다.

## 7. 이번 프로젝트에 적용할 추천 작업 방식

현재까지의 해석을 기준으로 하면 다음 방식이 가장 적절하다.

### 7.1 2024는 기준 디자인으로 사용

2024는 시험 플랫폼의 실제 토대다. 따라서 HTML 시안이나 향후 구현도 2024의 구조를 중심으로 잡아야 한다.

우선 MCP로 확인할 2024 기준 화면:

| 우선순위 | 화면 |
|---|---|
| 1 | `목록_시험접수` |
| 2 | `목록_시험응시` |
| 3 | `목록_시험결과` |
| 4 | `관리자 홈` |
| 5 | `회원관리` |
| 6 | `시험관리` |
| 7 | `출제관리` |
| 8 | `채점관리` |

### 7.2 2025는 UX/UI 참고로 사용

2025는 2024를 대체하는 화면 세트로 보기보다, 일부 UX/UI 패턴을 참고하는 자료로 보는 것이 안전하다.

특히 `창작작업실01.png`는 현재 기준에서 확정 메뉴가 아니라 참고 자료로 분리한다.

적용 가능한 부분:

- 작업형 화면의 좌우 분할
- 입력 영역과 결과 영역의 구분
- 파일 업로드 또는 템플릿 선택 흐름
- 사용자가 현재 작업 상태를 이해하기 쉬운 패널 구성

적용을 조심해야 하는 부분:

- 2024의 기본 사이트 구조를 덮어쓰는 것
- 창작작업실을 핵심 메뉴로 확정하는 것
- 시험접수/응시/결과 흐름보다 작업실 UX를 앞세우는 것

### 7.3 vibecoding은 관리자 고도화 기능으로 사용

vibecoding은 일반 사용자 화면보다 운영자용 고급 기능에 가깝다.

적용 가능한 부분:

- 대량 데이터 테이블
- 고급 검색 필터
- 파일 업로드
- 엑셀 다운로드
- SMS/메일 발송
- 다중 선택 후 일괄 처리

적용할 위치:

- 관리자 회원관리
- 관리자 시험관리
- 채점관리
- 결제/정산관리
- 문의/알림 관리

## 8. Figma 링크를 줄 때 가장 좋은 방식

Figma MCP로 정확히 분석하려면 링크를 줄 때 다음 방식이 좋다.

### 8.1 가장 좋은 링크

개별 화면 프레임을 선택한 뒤 복사한 링크가 가장 좋다.

예:

```text
https://www.figma.com/design/{fileKey}/{fileName}?node-id=5-27589
```

이 방식은 MCP가 어떤 화면을 봐야 하는지 명확하다.

### 8.2 페이지 전체 링크

페이지 전체 링크는 탐색용으로는 좋지만, 구현 기준으로는 애매할 수 있다.

이 경우에는 먼저 “이 페이지 안의 화면 목록을 파악해줘” 같은 목적으로 쓰는 것이 좋다. 이후 대표 화면을 다시 선택해서 개별 링크를 주는 방식이 더 정확하다.

### 8.3 여러 화면을 분석해야 할 때

한 번에 많은 링크를 주기보다, 다음 형식으로 묶어주는 것이 좋다.

| 순서 | 화면명 | Figma 링크 | 목적 |
|---|---|---|---|
| 1 | 시험접수 목록 | node 링크 | 마이페이지 목록 기준 |
| 2 | 시험응시 | node 링크 | 응시 상태/버튼 확인 |
| 3 | 관리자 홈 | node 링크 | 관리자 기본틀 확인 |
| 4 | 회원관리 | node 링크 | 관리자 목록 기준 |

이렇게 주면 분석 결과도 화면별로 정리되고, 구현 우선순위도 명확해진다.

## 9. MCP 분석 결과를 구현으로 바꾸는 방법

MCP에서 나온 코드를 그대로 복사해서 구현하는 것은 권장하지 않는다. Figma에서 생성되는 코드는 화면 재현에는 도움이 되지만, 실제 서비스 코드로는 중복이 많고 유지보수가 어렵기 때문이다.

더 좋은 방식은 다음과 같다.

1. MCP 결과에서 레이아웃 구조를 파악한다.
2. 색상, 간격, 폰트 크기, 테이블 구조 같은 기준값을 뽑는다.
3. 실제 구현에서는 공통 컴포넌트로 다시 만든다.
4. 화면별 데이터와 버튼 상태만 바꾼다.
5. HTML 시안으로 먼저 검증한 뒤, 필요할 때 Next.js나 React로 옮긴다.

이번에 만든 `시험플랫폼_레이아웃_통합도.html`도 이 방식에 맞춰야 한다. 즉, Figma의 2024 화면을 그대로 복사하는 것이 아니라, 2024의 화면 골격을 기준으로 사람이 이해하기 쉬운 통합 시안으로 재구성하는 것이 맞다.

## 10. 최종 권장 프로세스

전체 작업은 다음 순서가 가장 안정적이다.

| 단계 | 작업 | 산출물 |
|---|---|---|
| 1 | PDF/파일명으로 전체 화면 목록 정리 | 사이트맵, 기능 정의 |
| 2 | Figma MCP로 기준 화면 추출 | 기준 화면별 디자인 분석 |
| 3 | 공통 레이아웃과 컴포넌트 도출 | 헤더, 사이드바, 검색, 테이블, 버튼 정의 |
| 4 | 화면 유형별 템플릿 설계 | 목록형, 상세형, 작업형, 팝업형 |
| 5 | 2024/2025/vibecoding 역할 분리 | 2024 기본, 2025 참고, vibecoding 고도화 |
| 6 | HTML 시안 제작 | 빠르게 볼 수 있는 정적 레이아웃 |
| 7 | 실제 프론트엔드 구현 | Next.js/React 등으로 전환 |
| 8 | 기능 요구사항과 비교 | 누락 기능, 과해석 기능, 우선순위 정리 |

## 11. 이 프로젝트에서 특히 주의할 점

현재 분석에서 가장 중요한 주의점은 다음이다.

1. 2024 디자인 기본틀을 무시하면 안 된다.
2. 2025의 창작작업실은 확정 기능이 아니라 참고 자료로 둔다.
3. vibecoding은 일반 사용자 화면이 아니라 운영 고도화 기능으로 분류한다.
4. Figma MCP는 기준 화면을 정밀하게 확인하는 용도로 사용한다.
5. 전체 구현은 개별 화면 복사가 아니라 공통 템플릿화로 처리한다.

정리하면, Figma MCP는 “모든 화면을 자동으로 만들어주는 버튼”이 아니라 **기준 화면을 정확히 해석해서 전체 구현 체계를 흔들리지 않게 만드는 분석 도구**로 쓰는 것이 가장 좋다.
