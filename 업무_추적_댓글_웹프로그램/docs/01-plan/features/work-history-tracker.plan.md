## 1. Overview

### 1.1 Purpose

프로젝트와 부서 단위로 업무 내용을 정리하고, 하나의 화면에서 업무의 발생부터 진행, 논의, 결과까지 시간 순서로 추적할 수 있는 시스템을 구축한다. 각 업무에는 댓글과 대댓글 형태의 논의 기록 및 파일 첨부를 제공하여 업무 인수인계와 과거 의사결정 확인 비용을 줄인다.

### 1.2 Background

업무 진행 과정이 메신저, 파일, 구두 협의 등에 흩어지면 특정 업무가 어떤 배경으로 변경되었고 누가 무엇을 확인했는지 파악하기 어렵다. 업무 본문과 논의, 첨부자료를 동일한 맥락에 보존하는 추적 가능한 기록 화면이 필요하다.

### 1.3 Target Users

- 부서 구성원: 담당 업무 등록, 진행 기록 작성, 파일 공유, 댓글 참여
- 담당자 또는 협업자: 자신의 업무와 관련 논의 확인 및 후속 작업 처리
- 부서 관리자: 프로젝트 및 부서 업무 진행 상황과 히스토리 검토

## 2. Goals

### 2.1 Primary Goals

- [x] 프로젝트와 부서를 기준으로 업무 항목을 분류하고 조회할 수 있다.
- [x] 업무 상세 화면 한 페이지에서 본문, 상태 변화, 댓글, 대댓글, 첨부파일을 시간 흐름으로 확인할 수 있다.
- [x] 업무 관련 논의가 댓글 스레드로 연결되어 작성자와 작성 시점을 추적할 수 있다.
- [x] 업무 및 논의에 필요한 파일을 첨부하고 다시 확인할 수 있다.
- [x] 화면 여백을 줄인 정보 밀도 높은 레이아웃으로 다수의 업무 기록을 효율적으로 탐색할 수 있다.

### 2.2 Non-Goals

- 초기 범위에서 채팅, 화상회의, 이메일 발송 기능은 제공하지 않는다.
- 칸반, 간트 차트, 근태 또는 비용 정산 등 종합 ERP 기능은 포함하지 않는다.
- 외부 클라우드 드라이브와의 양방향 동기화는 초기 범위에서 제외한다.
- AI 요약, 자동 분류, 자동 일정 추천은 후속 검토 대상으로 둔다.

## 3. Scope

### 3.1 In Scope

- 프로젝트와 부서의 등록 또는 선택, 업무와의 연결
- 업무 등록 및 수정: 제목, 내용, 프로젝트, 부서, 담당자, 상태, 우선순위, 작성일/수정일
- 프로젝트별 또는 부서별 업무 목록, 필터 및 기본 검색
- 업무 상세 단일 페이지: 기본 정보, 본문, 첨부자료, 활동 히스토리, 댓글 스레드 표시
- 댓글과 1단계 이상 대댓글 작성, 수정 표시, 작성자와 작성 시각 표시
- 업무 생성/수정, 상태 변경, 댓글/대댓글 작성, 파일 첨부 이벤트의 히스토리 기록
- 업무 및 댓글 맥락에 연결된 파일 첨부, 다운로드, 파일명/크기/작성자/등록 시각 표시
- 컴팩트 UI: 좁은 여백, 고정 필터 또는 요약 영역, 스크롤 가능한 히스토리 영역
- 기본 접근 제어: 사용자의 열람 가능 부서 및 작성/수정 권한 고려

### 3.2 Out of Scope

- 조직 전체의 복잡한 결재선 및 전자결재 워크플로우
- 실시간 공동 문서 편집 및 실시간 채팅
- 파일 버전 비교, OCR, 문서 미리보기 고도화
- 모바일 네이티브 앱 및 오프라인 동기화
- 고급 대시보드, 업무량 예측, 통계 리포트

| FR-06 | 업무의 생성, 정보 변경, 상태 변경, 댓글 및 첨부 동작은 시간 순 히스토리에 남는다. | Must |

| FR-10 | 히스토리가 긴 업무에서도 최신/전체 활동을 쉽게 탐색할 수 있다. | Should |

## 5. Non-Functional Requirements

| Category | Requirement |
|----------|-------------|
| Usability | 목록과 상세 화면은 넓은 여백 대신 정보 중심의 컴팩트 레이아웃을 사용하고 주요 동작은 화면 이동을 최소화한다. |
| Traceability | 추적 대상 이벤트는 작성자, 발생 시각, 대상 업무, 동작 유형을 보존한다. |
| Security | 첨부 다운로드와 업무 열람은 사용자 권한 범위 안에서만 허용하고, 파일 업로드의 허용 형식/용량/악성 파일 대응 방침을 설계한다. |
| Performance | 일반 업무 상세의 본문과 초기 히스토리 목록은 사용자가 기다림을 체감하지 않는 수준을 목표로 하며, 긴 스레드는 페이지네이션 또는 점진 로딩을 고려한다. |
| Reliability | 첨부 업로드 실패와 댓글 저장 실패 시 사용자가 재시도할 수 있고 실패 상태를 분명히 표시한다. |
| Maintainability | 업무, 댓글, 첨부, 히스토리를 명확한 데이터 단위로 분리하여 이후 알림 또는 통계 기능 추가가 가능해야 한다. |

## 6. Initial Screen Concept

### 6.1 Core Pages

| Page | Purpose | Key Content |
|------|---------|-------------|
| 업무 목록 | 찾을 업무를 빠르게 좁히기 | 프로젝트/부서/담당자/상태 필터, 검색, 조밀한 테이블 또는 리스트 |
| 업무 상세 및 히스토리 | 업무 맥락과 진행 이력을 한 곳에서 추적 | 업무 요약, 첨부 영역, 활동 타임라인, 댓글/대댓글 작성 |
| 업무 등록/편집 | 업무 분류와 책임 정보 관리 | 제목, 내용, 프로젝트, 부서, 담당자, 상태, 우선순위, 첨부 |

### 6.2 Compact Layout Direction

- 데스크톱 상세 화면은 좌측 또는 상단에 업무 요약/첨부 정보를 압축 배치하고, 중심 영역에 시간 순 히스토리와 댓글 스레드를 우선 표시한다.
- 목록은 카드보다 조밀한 행 기반 표시를 우선 검토하여 한 화면에서 많은 업무를 비교할 수 있게 한다.
- 댓글 입력과 파일 추가 동작은 상세 페이지 안에서 제공하여 불필요한 페이지 전환을 줄인다.
- 긴 내용은 접기/펼치기 및 필터(전체, 변경, 댓글, 파일)를 통해 탐색성을 유지한다.

## 7. Success Criteria

- [ ] 프로젝트 및 부서를 지정한 업무를 생성하고 목록에서 해당 기준으로 찾을 수 있다.
- [ ] 하나의 업무 상세 화면에서 업무 내용, 상태 변경 이력, 댓글, 대댓글, 첨부파일 기록을 시간 순으로 확인할 수 있다.
- [ ] 댓글 작성과 대댓글 작성 후 작성자 및 작성 시간이 히스토리에 반영된다.
- [ ] 허용된 파일을 첨부하고 권한이 있는 사용자가 다운로드할 수 있다.
- [ ] 누가 언제 업무 정보를 변경했는지 핵심 변경 이벤트를 조회할 수 있다.
- [ ] 정보 밀도 높은 화면에서도 필터, 스레드 계층, 첨부 동작을 혼동 없이 사용할 수 있는지 검토한다.
- [ ] 설계 문서에서 데이터 모델, 권한 모델, 파일 저장/제한 정책, 히스토리 보존 정책, API 또는 데이터 접근 방식을 확정한다.

## 8. Risks & Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| 댓글, 변경 이력, 첨부 이벤트가 섞여 히스토리가 읽기 어려워짐 | High | Medium | 이벤트 유형별 시각 구분과 필터, 스레드 접기/펼치기를 설계한다. |
| 첨부파일로 인한 보안 또는 저장 용량 문제 | High | Medium | 접근 권한, 크기/확장자 제한, 스캔/저장 정책을 설계 단계에서 확정한다. |
| 부서별 접근 범위가 불명확하여 정보 노출 발생 | High | Medium | 역할과 부서 소속에 따른 열람/편집 권한 매트릭스를 먼저 정의한다. |
| 과도하게 조밀한 레이아웃으로 사용성이 저하됨 | Medium | Medium | 핵심 업무 흐름을 기준으로 최소 여백과 가독성의 균형을 목업에서 검증한다. |
| 긴 히스토리 및 대용량 첨부로 상세 화면 성능 저하 | Medium | Medium | 히스토리 점진 로딩, 첨부 메타데이터 분리, 인덱스/조회 전략을 설계한다. |

## 9. Architecture Considerations

- 핵심 데이터 후보는 `Department`, `Project`, `WorkItem`, `Comment`, `Attachment`, `ActivityEvent`, `User`이다.
- `Comment`는 부모 댓글 참조를 통해 대댓글 구조를 표현하고, `ActivityEvent`는 상태 변경 및 작성 행위를 일관되게 기록하는 방식을 검토한다.
- 첨부파일 원본 저장소와 첨부 메타데이터 저장 위치, 다운로드 권한 검증 경계를 설계 단계에서 결정한다.
- 초기 구현 기술 스택, 로그인 방식, 데이터베이스 및 배포 방식은 현재 프로젝트에 기존 구현이 없으므로 설계 전에 선택한다.

## 10. Open Decisions For Design

| Decision | Question |
|----------|----------|
| Authentication | 사용자는 어떻게 로그인하며 사내 사용자/부서 정보는 어디에서 관리할 것인가? |
| Authorization | 부서 간 공개 범위와 관리자/작성자/담당자별 편집 권한은 어떻게 구분할 것인가? |
| Thread Depth | 대댓글을 1단계로 제한할지, 다단계 스레드를 허용할지 결정한다. |
| Audit Policy | 수정 전후 값, 삭제된 댓글/첨부의 보존 및 표시 정책을 결정한다. |
| File Policy | 최대 용량, 허용 파일 형식, 스토리지, 다운로드/삭제 및 보안 검증 정책을 정한다. |
| Deployment | 내부 전용 서비스 여부와 배포/백업/접속 환경을 결정한다. |
