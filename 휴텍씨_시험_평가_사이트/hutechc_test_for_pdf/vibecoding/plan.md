# vibecoding Website 구현 계획

## 역할

`vibecoding`은 전체 서비스 버전이 아니라 `파일업로드`와 `구분/분류` 화면에 집중된 특화 모듈이다. 2025의 파일업로드 화면과 동명/동계열 자료가 많으므로 2025 신규 기능의 파생 또는 실험 시안으로 본다. 단, 파일이 완전히 동일하다고 단정하지 않고 화면 비교 후 공통화한다.

## 기준 자료

- 디자인 원본: `../../figma_pdf/vibecoding`
- 전체 계획: `../../plan.md`
- 비교 분석: `../../figma_pdf_2024_2025_vibecoding_비교분석.md`
- 2025 기준 계획: `../2025/plan.md`

## 구현 목표

- Next.js App Router 기반으로 파일업로드/구분 모듈 route group을 구성한다.
- 2024/2025 전체 IA와 분리된 기능 모듈로 설계한다.
- 2025 파일업로드 화면과 비교한 뒤 공통 레이아웃 재사용 범위를 결정한다.
- `구분.pdf`, `구분-1.pdf` ~ `구분-14.pdf`를 상태 또는 단계별 플로우로 정리한다.
- 파일 업로드 기반 대량 처리, 분류, 검수, 결과 확인 흐름은 `구분` PDF 매핑 후 확정한다.
- 모든 구현은 TDD 방식으로 작업한다.
- 구현 결과는 `../../figma_pdf/vibecoding`의 대응 PDF/이미지에 충실해야 하며, 2025 파일업로드 화면과 비교할 수 있어야 한다.

## TDD 작업 방식

vibecoding은 파일업로드/구분 처리 모듈이므로 상태 전이와 데이터 처리 흐름을 테스트로 먼저 고정한다.

작업 순서:

1. `구분` PDF들이 같은 화면의 상태 변화인지 단계별 플로우인지 분류한다.
2. PDF 매핑 결과로 확인된 업로드, 검증, 구분 처리, 검수, 저장 상태 모델 테스트를 먼저 작성한다.
3. 파일 업로드 컴포넌트와 결과 테이블 컴포넌트 단위 테스트를 작성한다.
4. PDF에서 확인된 구분 기준 선택과 일괄 선택/수정 액션 테스트를 작성한다.
5. 최소 구현 후 테스트를 통과시킨다.
6. vibecoding PDF/이미지와 구현 스크린샷을 비교한다.
7. 2025 `기본화면_파일업로드.jpg`와도 비교해 공통/차이를 기록한다.

우선 테스트 대상:

- 파일 업로드 성공/실패
- 파일 검증 오류 표시
- 구분 기준 선택
- 구분 처리 상태 전이
- 결과 테이블 체크박스 선택/일괄 선택
- 인라인 수정
- 저장/반영 완료와 실패
- 처리 이력 조회

## Figma 충실도 기준

각 화면은 `../../figma_pdf/vibecoding`의 대응 디자인을 기준으로 구현한다. 특히 `구분-1` ~ `구분-14`는 구현 후 비교를 위해 파일별 상태 또는 단계 매핑을 남긴다.

비교 항목:

- 파일업로드 기본화면의 레이아웃
- 구분 화면의 단계/상태별 변화
- 체크박스 위치, 선택 상태, 일괄 처리 영역
- 결과 테이블 컬럼과 편집 가능 셀
- 업로드/저장/반영 버튼 위치
- 처리 상태 배지와 오류 메시지
- 2025 파일업로드 화면과의 공통/차이

화면별 작업 기록에는 다음을 남긴다.

- 기준 vibecoding 디자인 파일
- 대응되는 2025 디자인 파일
- 구현 라우트
- 작성한 테스트
- 상태/단계 매핑
- PDF/이미지 대비 차이와 의도적 변경 여부

## 기능 IA 가설

아래 IA는 현재 파일명 기준의 가설이다. `구분.pdf`, `구분-1.pdf` ~ `구분-14.pdf`의 실제 화면 상태를 확인한 뒤 확정한다.

| 단계 | 화면 | 목적 | 확정 조건 |
| --- | --- | --- | --- |
| 1 | 파일업로드 기본화면 | 처리할 파일 선택 및 업로드 | `기본화면_파일업로드.jpg` 기준 |
| 2 | 업로드 파일 확인 | 파일명, 형식, 오류 여부 확인 | `구분` PDF 중 업로드 결과 화면 확인 필요 |
| 3 | 구분 기준 선택 | 대분류/중분류/소분류 또는 처리 조건 선택 | `구분` PDF 중 분류 기준 UI 확인 필요 |
| 4 | 구분 처리 | 업로드 데이터를 기준에 따라 분류 | 처리 진행/완료 상태 화면 확인 필요 |
| 5 | 구분 결과 검수 | 결과 테이블 확인, 체크박스 선택, 수정 | 결과 테이블과 편집 가능 셀 확인 필요 |
| 6 | 저장/반영 | 최종 결과 저장 또는 관리자 데이터에 반영 | 저장/반영 버튼과 완료 화면 확인 필요 |
| 7 | 처리 이력 | 과거 업로드와 처리 결과 조회 | 실제 이력 화면이 있을 때만 라우트 확정 |

## 2025와의 관계

- `기본화면_파일업로드.jpg`는 2025와 vibecoding에 모두 존재하는 동명/동계열 화면이다. 시각 비교 후 공통 `FileUploadDropzone`과 `UploadedFileList` 사용 여부를 결정한다.
- `구분-4.pdf`, `구분-5.pdf`는 2024/2025/vibecoding에 모두 있으므로 분류/구분 공통 패턴 후보로 둔다.
- vibecoding의 `구분` 화면들은 2025의 파일업로드 후속 단계로 붙이는 것이 가장 안전하다.

## 상태 모델

| 도메인 | 상태 |
| --- | --- |
| 업로드 | 대기, 업로드중, 업로드완료, 업로드실패 |
| 검증 | 검증대기, 검증중, 정상, 오류있음 |
| 구분 처리 | 처리대기, 처리중, 처리완료, 처리실패 |
| 검수 | 미검수, 검수중, 수정필요, 검수완료 |
| 반영 | 저장대기, 저장완료, 반영완료, 반영실패 |

상태값은 PDF 매핑 전 초안이다. 실제 화면에 없는 상태는 구현 범위에서 제외한다.

## 확인 필요 사항

- `구분-1` ~ `구분-14`가 같은 화면의 상태 변화인지 단계별 플로우인지 확인
- 구분 기준이 시험 분류코드와 연결되는지, 별도 파일 처리 규칙인지 확인
- 업로드 대상 파일 형식과 최대 크기 확인
- 처리 결과가 어느 관리자 도메인에 반영되는지 확인
- 자동 구분 결과를 사용자가 직접 수정할 수 있는지 확인
