접근성 개선 전 / 후 비교

같은 UI를 접근성 미흡 상태와 개선 상태로 나란히 배치했습니다. 오른쪽 설명에서 왜 이렇게 바꿨는지 확인할 수 있습니다.

개선 전

이미지 대체텍스트 없음

<img src="exam-card.png" alt="">

정보성 이미지인데 alt가 비어 있어 스크린리더 사용자는 이미지가 전달하는 내용을 알 수 없습니다.

개선 후

의미를 설명하는 대체텍스트 제공

<img src="exam-card.png"
  alt="시험 접수 카드에 대표 이미지와 시험 정보가 함께 표시된 예시">

왜: 이미지를 볼 수 없는 사용자도 동일한 정보에 접근해야 하므로, 이미지의 장식이 아니라 기능과 의미를 설명합니다.

개선 전

색상만 다른 필터 버튼

<button type="button" class="on">접수중</button>

시각적으로만 선택 상태가 표시되고, 보조기술에는 선택 여부가 전달되지 않습니다.

개선 후

상태 속성과 텍스트를 함께 제공

<button type="button" aria-pressed="true">
  접수중 선택됨
</button>

왜: 토글형 버튼은 aria-pressed로 상태를 전달해야 키보드/스크린리더 사용자가 현재 필터 상태를 알 수 있습니다.

개선 전

명도 대비가 부족한 버튼

background: #0088ff;
color: #ffffff;

흰색 글자와 밝은 파란색 배경 조합은 일반 텍스트 기준 4.5:1에 못 미칠 수 있습니다.

개선 후

대비를 확보한 버튼

background: #006dcc;
color: #ffffff;

왜: 버튼 텍스트도 정보이므로 배경과 충분한 대비를 가져야 저시력 사용자와 밝은 환경 사용자 모두 읽을 수 있습니다.

개선 전

입력 오류를 색상만으로 표시

다시 입력

<input class="error">
<p>다시 입력</p>

오류 원인과 수정 방법이 불명확하고 입력과 오류 문구가 연결되어 있지 않습니다.

개선 후

오류 원인과 수정 방법 연결

이메일 형식으로 입력해야 합니다. 예: name@example.com

<input aria-invalid="true"
  aria-describedby="after-email-error">

왜: 오류 위치, 원인, 해결 방법이 입력 요소와 연결되어야 사용자가 무엇을 고쳐야 하는지 바로 알 수 있습니다.

접근성 확인 모달

모달 제목과 설명이 연결되어 있고, 키보드 초점이 모달 안에서 관리되어야 합니다.

대체텍스트 · 이미지 접근성

이미지가 전달하는 정보에 따라 alt 제공 방식이 달라집니다. 의미 있는 이미지는 목적을 설명하고, 장식 이미지는 보조기술에서 제외합니다.

의미 있는 이미지

정보를 전달하는 이미지는 화면에 보이는 핵심 의미를 alt에 적습니다.

<img src="exam-card.png"
  alt="시험 접수 카드에 대표 이미지와 시험 정보가 함께 표시된 예시">

장식 이미지

문맥상 의미가 없고 꾸밈 역할만 하는 이미지는 alt="" 또는 aria-hidden="true"로 읽히지 않게 합니다.

<img src="pattern.png" alt="">
<span class="decoration" aria-hidden="true"></span>

아이콘 버튼

아이콘만 있는 버튼은 보이는 텍스트가 없으므로 aria-label로 기능을 제공합니다.

<button type="button" aria-label="검색 조건 초기화">
  <span aria-hidden="true">↺</span>
</button>

버튼 상태 · 선택 상태 · 비활성 상태

버튼은 기본, hover, focus, pressed, disabled 상태를 시각적으로 구분하고 보조기술에 필요한 상태 값을 전달해야 합니다.

상태 필수 구현 접근성 기준
기본 버튼 목적을 텍스트로 표시합니다. 예: 검색, 신청 저장, 필터 전체 해제 버튼 이름이 스크린리더에 전달되어야 합니다.
초점 :focus-visible 윤곽선을 제공하고 배경색 변화만 의존하지 않습니다. 키보드 사용자가 현재 위치를 즉시 볼 수 있어야 합니다.
선택/토글 aria-pressed="true|false" 또는 aria-selected="true|false"를 상태에 맞게 갱신합니다. 색상 외에 선택 상태가 프로그램적으로 전달되어야 합니다.
비활성 disabled를 사용하고, 비활성 이유를 주변 문구로 안내합니다. 실행 불가 상태가 명확해야 하며 초점 순서가 혼란스럽지 않아야 합니다.

명도 대비 · 색상 점검

일반 텍스트는 4.5:1 이상, 18pt 이상 또는 14pt 굵은 큰 텍스트는 3:1 이상, UI 경계와 아이콘은 3:1 이상을 기준으로 점검합니다.

본문 텍스트 #222222 / #FFFFFF 약 15.9:1 · 통과
보조 텍스트 #666666 / #FFFFFF 약 5.7:1 · 통과
접근성 버튼 #FFFFFF / #006DCC 약 5.2:1 · 통과
주의 조합 #FFFFFF / #0088FF 약 3.5:1 · 일반 텍스트 주의
적용 기준: 접근성 점검 페이지에서는 primary 버튼과 선택 필터를 #006DCC로 보정했습니다. 기존 토큰 #0088FF 위 흰색 텍스트는 큰 글자나 굵은 UI 라벨이 아닌 일반 본문 크기에서는 대비 재검토가 필요합니다.

웹접근성 점검 체크리스트

한국형 웹 콘텐츠 접근성 지침(KWCAG)의 큰 원칙과 WCAG의 POUR 원칙을 실제 컴포넌트 기준으로 정리했습니다.

1. 인식의 용이성

  • 의미 있는 이미지는 대체 텍스트를 제공하고, 장식 이미지는 aria-hidden="true"로 숨깁니다.
  • 아이콘 버튼처럼 텍스트가 없는 컨트롤은 aria-label로 이름을 제공합니다.
  • 텍스트와 배경의 명도 대비는 일반 텍스트 4.5:1 이상, 큰 텍스트 3:1 이상을 목표로 합니다.
  • 상태는 색상만으로 전달하지 않고 "접수중", "마감", "오류" 같은 텍스트를 함께 표시합니다.
  • 표에는 caption, th, scope를 사용해 제목과 데이터 관계를 제공합니다.

2. 운용의 용이성

  • 모든 링크, 버튼, 입력 요소는 키보드 Tab, Enter, Space로 접근 가능해야 합니다.
  • 초점 위치는 :focus-visible 스타일로 명확히 보여야 합니다.
  • 반복 내비게이션 앞에는 본문 바로가기 링크를 제공해 키보드 사용자의 이동 부담을 줄입니다.
  • 클릭 대상은 충분한 크기와 간격을 가져야 하며 모바일에서도 오작동을 줄여야 합니다.

3. 이해의 용이성

  • 문서 언어는 html lang="ko"로 선언합니다.
  • 입력 필드는 보이는 label과 연결하고, 필수 여부와 도움말을 명확히 제공합니다.
  • 오류는 색상만으로 표시하지 않고 원인과 수정 방법을 문장으로 안내합니다.
  • 버튼 문구는 "확인"만 반복하지 말고 가능한 경우 "신청 저장", "필터 전체 해제"처럼 결과를 드러냅니다.

4. 견고성

  • HTML은 의미 요소를 우선 사용하고, 필요한 경우에만 ARIA를 보조로 사용합니다.
  • 탭, 토글, 모달처럼 상태가 바뀌는 컴포넌트는 aria-selected, aria-pressed, aria-modal을 갱신합니다.
  • 동적 알림은 role="status" 또는 aria-live 영역으로 보조기술에 전달합니다.
  • 브라우저 확대 200%와 반응형 화면에서도 내용이 겹치거나 잘리지 않아야 합니다.
