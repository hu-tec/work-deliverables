## 1. 통합 목적

4차 작업은 1차~3차 생성의 누락을 다시 반복하지 않기 위해 하나의 기준 문서로 진행한다. 2024 자료는 전체 시험 운영 흐름, 2025 자료는 신규 AI 작업/양식/템플릿/전문가/관리자 개편 화면, vibecoding과 `DB페이지_v2.html`은 관리자 고밀도 대량관리 UI와 DB 샘플 데이터 기준으로 사용한다.

핵심 목표는 다음과 같다.

- 2024 전체 시험 운영 플로우를 빠짐없이 연결한다.
- 2025 figma_pdf 기능을 빠뜨리지 않는다.
- 디자인 구성을 최대한 반영한다.
- 모듈 사이 간격을 좁게 잡고 고밀도 관리자 UI를 구현한다.
- 글자 크기, 버튼, 탭, 테이블, 폼을 4단 그리드 기준으로 정렬한다.
- DB가 주어진 경우 해당 DB의 필터, 컬럼, 샘플 데이터를 화면에 반영한다.
- 사용자 플로우에서 빠진 내용을 점검하고 보완한다.

## 3. 합친 항목 설명

- 1차의 `2025 신규/개편 화면 범위`를 2차/3차의 전체 역할별 업무 IA 안에 넣었다.
- 1차의 `양식관리`, `템플릿`, `창작작업실`, `메타번역`, `파일업로드`, `장바구니`, `결제`를 4차 핵심 신규 기능으로 승격했다.
- 2차/3차의 공개 홈페이지, 인증, 수험생, 출제자, 채점자, 관리자 플로우를 4차의 기본 골격으로 통합했다.
- 2차/3차의 `vibecoding 대량관리` 요구를 현재 `DB페이지_v2.html`의 실제 UI/DB 항목과 합쳤다.
- 3차의 `figma_pdf 보완 필요사항`은 누락 또는 임시 판단으로 두지 않고 4차 구현 시 필요한 상태값/오류/정책 체크리스트로 편입했다.
- `figma_pdf_2025_사이트맵_기능정의.md`의 목차와 사용자 플로우는 그대로 반영하되, 창작작업실과 메타-T 번역은 “참고”가 아니라 4차에서는 실제 개발 대상에 포함한다.
- DB는 임의 문자열 대신 현재 HTML에서 확인된 필터, 컬럼, 15개 샘플 행을 우선 반영하고, 추후 CSV/XLSX가 오면 같은 모델에 추가한다.

## 4. 4차 구현 핵심 원칙

- 첫 화면은 사용 가능한 서비스 화면이어야 하며, 설명용 랜딩에 머물면 안 된다.
- 관리자 화면은 `DB페이지_v2.html`처럼 좌측 60px 아이콘 레일, 상단 56px 헤더, 20px 내외 콘텐츠 패딩, 고밀도 테이블 구조를 따른다.
- 모듈 간 간격은 일반 웹사이트보다 좁게 잡는다. 관리자/운영 화면 기준 섹션 간격은 12~16px, 폼 행 간격은 6~8px, 테이블 셀 패딩은 4~8px로 제한한다.
- 4단 그리드는 모든 대시보드/폼/카드/검색필터에 적용한다.
- 화면에는 실제 기능 조작 요소를 배치한다. 버튼만 있는 더미가 아니라 탭 전환, 역할 전환, 필터 검색, 업로드 상태, 결제/저장/제출 확인이 동작해야 한다.
- `figma_pdf`에서 확인되는 기능은 이름이 화면명으로만 있어도 개발 대상에 포함한다.

## 5. 디자인 시스템 및 UI/UX 기준

`DB페이지_v2.html`에서 확인된 디자인 기준을 4차 기본 스타일로 사용한다.

| 항목 | 기준 |
|---|---|
| 기본 배경 | `#f2f3f7` |
| 패널 배경 | `#ffffff` |
| 주요 선 | `#e3e5ed`, `#ddd`, `#eee` |
| 주요 텍스트 | `#222` |
| 보조 텍스트 | `#808080`, `#999` |
| 포인트 컬러 | `#0088ff`, hover `#006fcc` |
| 위험/환불 | `#dc3d3d` |
| 진행/전문가 강조 | `#7c3aed` |
| 라운드 | 버튼/패널 4~6px, 카드 6~8px 이하 |
| 기본 본문 | 12~14px |
| 관리자 제목 | 20~22px |
| 테이블 셀 | 12px, 줄간격 1.35~1.45 |
| 버튼 높이 | 28~36px |
| 체크박스 | 16px, 라벨 간격 4px |

사용자 화면의 창작작업실/메타-T 번역/결제는 관리자보다 여백을 조금 넓힐 수 있지만, 4차 요구에 맞춰 전체 모듈 간격은 과하게 벌리지 않는다.

## 6. 4단 그리드 적용 기준

4단 그리드는 화면 너비와 무관하게 “업무 정보 덩어리”를 4개 기준으로 정리하는 규칙이다.

| 사용처 | 적용 |
|---|---|
| 대시보드 요약 | 신규 회원, 접수 현황, 결제 현황, 채점 대기 4개 카드 |
| 검색 필터 | 한 행에 4개 필터 그룹을 우선 배치하고 좁은 화면에서는 2단/1단으로 축소 |
| 폼 입력 | 기본정보, 일정/상태, 분류/권한, 결제/옵션 4개 컬럼 |
| 작업 화면 | 설정, 입력, 결과, 액션/요약 4영역 |
| 테이블 상단 | 탭, 필터, 액션, 저장/다운로드 4블록 |
| 사용자 구매 | 상품, 옵션, 가격, 결제 CTA 4영역 |

CSS 구현 기준은 `grid-template-columns: repeat(4, minmax(0, 1fr))`이며, 1024px 이하에서는 2단, 640px 이하에서는 1단으로 접는다.

## 20. 사용자 플로우 누락 점검

| 플로우 | 상태 | 보완 기준 |
|---|---|---|
| 공개 홈페이지 -> 로그인 | 포함 | CTA와 로그인 진입 유지 |
| 로그인 -> 역할별 홈 | 포함 | 역할 변경 시 메뉴 재구성 |
| 수험생 시험 접수 -> 결제 -> 응시 -> 결과 -> 자격증 | 포함 | 취소/환불/다음차시/채점중 상태 포함 |
| 출제자 문제 작성 -> 미리보기 -> 제출 -> 반려 수정 | 포함 | 번역형/프롬프트형/단답형 모두 포함 |
| 채점자 배정 -> 채점 -> 평가서 -> 완료 | 포함 | 임시저장과 완료 후 수정 정책 필요 |
| 관리자 회원/전문가 승인 | 포함 | 승인/반려/보류/재심사 상태 전이 필요 |
| 관리자 시험 등록 -> 일정 -> 문제 연결 -> 결과 공개 | 포함 | 환불 규칙과 결과 공개일 필요 |
| 결제 -> 정산 -> 환불 | 포함 | 상태 전이와 포인트/구독/견적 요청 포함 |
| 자격증 신청 -> 발급 -> 재발급 | 포함 | 발급 번호와 파일 관리 포함 |
| 양식/템플릿 등록 -> 사용자 작업 화면 노출 | 포함 | 미리보기와 게시/비게시 필요 |
| 창작작업실 -> AI 처리 -> 편집 -> 장바구니/결제 | 포함 | 4영역 작업화면으로 연결 |
| 메타-T 번역 -> 비교 번역 -> 에디터 -> 저장/결제 | 포함 | AI별 결과와 변경 이력 필요 |
| DB 대량관리 -> 필터 -> 엑셀/파일/SMS/메일 | 포함 | 업로드 성공/실패/처리결과 필요 |

현재 사용자가 준 요구 기준으로 큰 플로우 누락은 없다. 다만 실제 운영 정책으로 확정해야 하는 세부 규칙은 21장에 남긴다.

## 21. figma_pdf 보완 필요사항 반영

3차 보완 문서의 항목은 4차에서 다음처럼 처리한다.

| 보완 항목 | 4차 처리 |
|---|---|
| 권한 정책 | 역할별 메뉴/버튼 숨김, 읽기/수정 범위 구현 |
| 회원/전문가 상세 | 기본정보, 활동, 결제, 문의, 자격증, 승인 항목 패널 구현 |
| 전문가 승인 절차 | 승인/반려/보류/재심사/관리자 메모 상태 구현 |
| 시험 등록 정책 | 시험명, 회차, 등급, 일정, 접수기간, 응시료, 환불 규칙 필드 구현 |

| 결제/정산/환불 | 결제완료, 미결제, 환불, 구독중, 포인트, 견적 요청 구현 |
| 자격증 발급 | 신청, 입금대기, 제작중, 배송중, 발급완료, 재발급 구현 |
| 양식/템플릿 | 양식 스키마, 템플릿, 추천 서식, 게시 상태 구현 |

| 알림/로그 | SMS/메일 액션과 변경 이력 영역 구현 |
| 빈 상태/오류 상태 | 데이터 없음, 권한 없음, 저장 실패, 업로드 실패, 결제 실패 구현 |
