# 일반인용 문서 읽기 가이드

작성일: 2026-05-21  
목적: 지금까지 만든 분석 문서들이 개발자 중심으로 느껴질 수 있는지 점검하고, 비전문가가 이해하기 쉬운 읽는 순서와 보완 필요 내용을 정리한다.

## 1. 먼저 결론

현재 문서들은 자료 분석, 화면 목록 정리, 기능 정의, 비교 분석에는 충분하다. 하지만 일반인이 처음 읽기에는 다음 이유로 어렵게 느껴질 수 있다.

| 어려운 점 | 이유 |
|---|---|
| 파일명과 화면명이 많이 나온다 | `출제관리_프롬프트상세.pdf`, `Group 758531575.jpg`처럼 내부 자료명이 계속 등장한다. |
| 개발/기획 용어가 많다 | IA, 엔티티, 권한, 데이터 모델, 구현 우선순위 같은 표현이 많다. |
| 전체 서비스 그림보다 세부 기능이 먼저 나온다 | 일반인은 “이 사이트가 무엇을 하는 곳인지”를 먼저 알고 싶어 한다. |
| 사용자 입장의 하루 흐름이 부족하다 | 수험생, 출제자, 채점자, 관리자 각각이 실제로 어떤 순서로 쓰는지 더 쉽게 보여줄 필요가 있다. |
| 화면을 직접 보지 않은 사람에게는 추상적이다 | 문서만 보면 버튼, 탭, 표가 어떤 느낌인지 상상하기 어렵다. |

따라서 일반인에게 설명할 때는 **파일 분석 문서부터 보여주면 안 된다.** 먼저 “이 서비스는 누구를 위한 서비스이고, 각 사용자가 무엇을 하는지”부터 설명해야 한다.

## 2. 일반인에게 이 서비스를 설명하면

이 서비스는 크게 보면 **시험 운영 플랫폼**이다. 2025 자료에는 AI 작업실처럼 보이는 참고 화면도 있지만, 현재 기준에서는 확정된 핵심 기능이 아니라 UX/UI 참고 자료로 보는 것이 맞다.

쉽게 말하면 다음 네 가지가 합쳐진 서비스다.

| 쉬운 이름 | 실제 의미 |
|---|---|
| 시험 신청 사이트 | 수험생이 시험을 접수하고 결제하고 응시한다. |
| 문제 출제 사이트 | 전문가나 출제자가 시험 문제를 만든다. |
| 채점 관리 사이트 | 채점자가 답안을 보고 점수를 매긴다. |
| 관리자 업무 시스템 | 운영자가 회원, 시험, 결제, 문의, 통계, 파일을 관리한다. |

2025 자료에는 다음과 같은 참고 화면이 함께 들어 있다.

| 쉬운 이름 | 실제 의미 |
|---|---|
| 작업형 UX/UI 참고 화면 | 사용자가 문서, 번역, 유튜브, 리서치 같은 작업을 처리하는 화면 구성을 참고한다. 실제 구현 기능인지는 별도 확정이 필요하다. |

`vibecoding` 자료는 별도 서비스라기보다, 관리자가 엑셀처럼 많은 데이터를 한 번에 보고 처리하는 **고급 관리자 화면**으로 이해하면 된다.

## 4. 역할별 쉬운 설명

### 4.1 수험생

수험생은 시험을 보는 사람이다.

수험생의 흐름은 다음과 같다.

1. 회원가입 또는 로그인한다.
2. 응시할 시험을 찾는다.
3. 시험을 접수한다.
4. 응시료를 결제한다.
5. 시험일에 시험을 시작한다.
6. 답안을 제출한다.
7. 시험 결과를 확인한다.
8. 합격하면 자격증을 신청한다.
9. 문제가 있으면 1:1 문의를 남긴다.

수험생에게 가장 중요한 화면은 “지금 내가 무엇을 해야 하는지”를 알려주는 화면이다. 예를 들어 결제를 안 했으면 `결제하기`, 시험 시간이 되었으면 `시험시작`, 합격했으면 `자격증신청`이 바로 보여야 한다.

### 4.2 출제자

출제자는 시험 문제를 만드는 사람이다.

출제자의 흐름은 다음과 같다.

1. 로그인한다.
2. 자신에게 배정된 시험이나 문제 유형을 확인한다.
3. 새 문제를 작성한다.
4. 정답, 해설, 평가 기준을 입력한다.
5. 임시저장하거나 미리보기로 확인한다.
6. 최종 제출한다.
7. 관리자가 반려하면 수정해서 다시 제출한다.
8. 승인되면 문제은행이나 시험에 반영된다.

출제자에게 중요한 것은 “문제를 잘못 제출하지 않도록 돕는 것”이다. 그래서 단계별 작성, 미리보기, 임시저장, 반려 사유 확인이 중요하다.

### 4.3 채점자

채점자는 수험생 답안을 평가하는 사람이다.

채점자의 흐름은 다음과 같다.

1. 로그인한다.
2. 자신에게 배정된 채점 목록을 확인한다.
3. 답안을 하나씩 열어본다.
4. 문제와 평가 기준을 확인한다.
5. 점수와 코멘트를 입력한다.
6. 임시저장하거나 채점완료 처리한다.
7. 필요하면 재검토 또는 피드백을 남긴다.

채점자에게 중요한 것은 “빠르지만 공정하게 채점하는 것”이다. 그래서 답안, 평가 기준, 점수 입력란이 한 화면에서 잘 보여야 한다.

### 4.4 관리자

관리자는 전체 서비스를 운영하는 사람이다.

관리자의 흐름은 다음과 같다.

1. 오늘 처리해야 할 업무를 대시보드에서 확인한다.
2. 회원 가입/승인/정지 상태를 관리한다.
3. 시험 일정과 접수 상태를 관리한다.
4. 출제된 문제와 채점 상태를 확인한다.
5. 결제, 환불, 정산을 확인한다.
6. 문의에 답변한다.
7. 공지, FAQ, 블로그, 배너를 관리한다.
8. 통계를 확인한다.
9. 필요하면 엑셀 파일로 데이터를 올리거나 내려받는다.

관리자에게 중요한 것은 “많은 정보를 안전하게 빠르게 처리하는 것”이다. 그래서 테이블, 필터, 엑셀, 파일 업로드, 권한 관리가 중요하다.

### 4.5 작업형 참고 화면 사용자

2025 자료에 새로 보이는 참고 화면을 기준으로 가정한 사용자다. 현재는 실제 서비스 사용자로 확정하기보다 UX/UI 참고 시나리오로만 보는 것이 안전하다.

작업형 참고 화면의 흐름은 다음과 같다.

1. 작업형 화면에 들어간다고 가정한다.
2. 문서, 유튜브, 번역, 리서치 같은 작업 유형을 고른다.
3. 파일이나 텍스트를 입력한다.
4. 사용할 AI를 선택한다.
5. 필요한 옵션을 선택한다. 예: 전문가 검수, 긴급 작업.
6. 결과를 확인하고 수정한다.
7. 장바구니에 담거나 결제한다.

이 사용자는 시험을 보는 사람이 아니라, 작업물을 만드는 화면 패턴을 설명하기 위한 가정 사용자다.

## 5. 일반인이 더 알고 싶어 할 내용

현재 문서에 추가되면 좋은 일반인용 설명은 다음과 같다.

| 필요한 설명 | 이유 |
|---|---|
| 실제 사용자 시나리오 | “홍길동 수험생이 시험을 접수하는 과정”처럼 예시가 있으면 이해가 빠르다. |
| 화면 용어 사전 | 접수, 응시, 출제, 채점, 정산, 검수, 양식, 템플릿 같은 단어를 쉽게 풀어야 한다. |
| 역할별 첫 화면 예시 | 수험생/출제자/채점자/관리자가 로그인 후 무엇을 보게 되는지 설명 필요 |
| 서비스 전체 흐름 그림 | 시험 흐름과 2025 참고 UX/UI 흐름을 구분해서 설명하는 다이어그램이 있으면 좋다. |
| 일반 사용자와 관리자의 차이 | 사용자가 보는 화면과 관리자가 보는 화면은 완전히 다르다는 점을 먼저 설명해야 한다. |
| 결제와 환불 흐름 | 돈이 오가는 부분은 일반인이 가장 궁금해하는 영역이다. |
| 개인정보 보호 방식 | 관리자 화면에 개인정보가 많기 때문에 어떻게 보호되는지 설명이 필요하다. |
| 오류 상황 안내 | 시험 시간이 지났을 때, 결제가 실패했을 때, 파일 업로드가 실패했을 때 사용자가 무엇을 해야 하는지 설명 필요 |

## 6. 용어를 쉬운 말로 바꾸기

기존 문서에서 많이 쓰는 용어는 일반인에게 다음처럼 바꿔 설명하는 것이 좋다.

| 기존 표현 | 쉬운 표현 |
|---|---|
| IA | 사이트 구조, 메뉴 구조 |
| 엔티티 | 관리해야 하는 정보 종류 |
| 데이터 모델 | 어떤 정보를 저장할지 정한 설계 |
| 권한 | 누가 어떤 화면을 볼 수 있는지 정한 규칙 |
| 상태값 | 현재 진행 상태 |
| 플로우 | 사용자가 일을 처리하는 순서 |
| 컴포넌트 | 반복해서 쓰는 화면 부품 |
| 고밀도 테이블 | 엑셀처럼 정보가 아주 많은 표 |
| 마스킹 | 개인정보 일부를 가려서 보여주는 것 |
| 검수 | 제출된 내용을 확인하고 승인/반려하는 것 |
| 정산 | 결제된 금액을 계산하고 지급/보고하는 일 |
| 양식 | 사용자가 입력해야 하는 항목 묶음 |
| 템플릿 | 미리 만들어둔 예시 양식 |

## 7. 일반인용으로 문서를 더 좋게 만드는 방법

현재 문서를 그대로 일반인에게 전달하기보다, 앞에 다음 자료를 하나 더 붙이면 좋다.

| 추가 자료 | 내용 |
|---|---|
| 1페이지 요약 | 이 서비스가 무엇인지, 누가 쓰는지, 핵심 기능 5개 |
| 역할별 사용 이야기 | 수험생, 출제자, 채점자, 관리자 중심의 하루 흐름. 2025 작업형 화면은 참고 시나리오로 분리 |
| 쉬운 용어 사전 | 개발/기획 용어를 쉬운 말로 바꾼 표 |
| 화면별 목적 표 | 각 화면이 “누가, 무엇을 하기 위한 화면인지”만 정리 |
| FAQ | 일반인이 물어볼 질문과 답변 |

## 9. 최종 판단

현재 만들어진 문서들은 개발자만을 위한 문서는 아니지만, **기획자와 개발자가 화면 범위를 맞추기 위한 문서에 더 가깝다.**

일반인에게 바로 보여주려면 다음 순서가 더 적합하다.

1. 이 문서로 전체 서비스를 쉽게 이해한다.
2. `2024_figma_사이트맵_기능정의.md`로 기본 시험 플랫폼을 본다.
3. `figma_pdf_2025_사이트맵_기능정의.md`로 2025 UX/UI 참고 화면과 양식/번역 화면을 본다.
