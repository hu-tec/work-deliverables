# vibecoding PDF와 ai_docs IA 비교 분석

작성일: 2026-05-21  
분석 대상:

- `figma_pdf/vibecoding/*.pdf`
- `figma_pdf/vibecoding/기본화면_파일업로드.jpg`
- `figma_pdf/vibecoding/Frame 758532606.jpg`
- `ai_docs/ia분석파일.docx`

주의: `vibecoding`의 PDF는 Figma에서 내보낸 화면 PDF라 내부 텍스트가 일반 문서처럼 안정적으로 추출되지 않았다. 따라서 이 문서는 PDF 파일명, PDF 메타데이터, 함께 포함된 대표 JPG 시안, 화면에서 식별 가능한 UI 구성을 기준으로 분석했다. `ai_docs`는 DOCX 내부 텍스트가 추출되어 IA 문서 내용 기준으로 비교했다.

## 결론 요약

`figma_pdf/vibecoding`은 전체 시험 플랫폼 설계서라기보다, 관리자 화면 안에서 대량 데이터를 조회하고 파일을 올려 처리하는 특정 업무 화면 묶음에 가깝다. 대표 화면은 `회원관리` 화면이며, 매우 많은 필터, 체크박스, 검색 조건, 엑셀 생성, 파일 업로드, 파일 다운로드, 가로 스크롤 테이블을 포함한다.

반면 `ai_docs/ia분석파일.docx`는 수험생, 출제자, 채점자, 관리자 전체 역할을 포함한 서비스 IA 문서다. 메뉴 구조, 역할별 페이지, 업무 흐름, 상태값, 권한, 보안, 개인정보 마스킹, 일괄 처리, 다운로드 기능까지 넓게 정의한다.

쉽게 말하면 `vibecoding PDF`는 "실제 화면 예시"에 가깝고, `ai_docs`는 "서비스 전체 설계도"에 가깝다.

## vibecoding PDF 분석

### 파일 구성

`vibecoding` 폴더에는 PDF 16개, JPG 2개, 체크박스 PNG 46개가 있다.

주요 파일은 다음과 같다.

| 파일 | 해석 |
|---|---|
| `제목 없음.pdf` | PDF 메타데이터상 `시험-바이브 진행` 화면. 용량이 약 14MB로 가장 크며, 전체 시안 묶음 또는 대형 화면 캡처로 보인다. |
| `구분.pdf`, `구분-1.pdf` ~ `구분-14.pdf` | `구분`이라는 같은 화면 또는 컴포넌트의 상태별/단계별 시안으로 보인다. |
| `기본화면_파일업로드.jpg` | 관리자형 회원관리 화면. 검색, 필터, 테이블, 저장, 파일 업로드/다운로드 기능이 보인다. |
| `Frame 758532606.jpg` | 매우 넓은 데이터 테이블 화면. 여러 관리 목록 또는 전체 컬럼을 한눈에 펼친 참고 이미지로 보인다. |
| `CheckBox*.png` | 화면 내 필터 체크박스 리소스. 체크/미체크/비활성 등 상태 표현용으로 보인다. |

### 대표 화면 설명

`기본화면_파일업로드.jpg`는 일반 사용자에게는 "관리자가 회원과 시험 관련 정보를 한 화면에서 검색하고 관리하는 페이지"로 이해하면 된다.

화면에는 다음 요소가 있다.

| 영역 | 설명 |
|---|---|
| 좌측 메뉴 | 홈, 사용자, 교육/시험, 설정 등으로 이동하는 아이콘 메뉴 |
| 상단 검색 | 전체 검색어를 입력하는 검색창 |
| 상단 사용자 영역 | `관리자`, `로그아웃` 표시 |
| 현재 위치 | `홈 > 회원관리` 형태의 breadcrumb |
| 탭 메뉴 | 전체, 회원관리, 활동관리, 결제관리, 1:1문의 관리, 통계 |
| 필터 영역 | 회원 유형, 성별, 프리미엄 여부, 지역, 연락 수단, 요청 상태 등 매우 많은 체크박스 |
| 검색 조건 | 카테고리 선택, 이름/검색어 입력, 날짜 범위, 검색 버튼 |
| 본문 테이블 | 회원 정보와 시험/결제/접수/채점/피드백 관련 정보를 한꺼번에 표시 |
| 파일 기능 | 엑셀 파일 생성, 파일 올리기, 파일 다운로드 |
| 일괄 기능 | 선택 항목 삭제, 메일 보내기, SMS 전송 |

### 화면의 핵심 성격

이 화면은 단순한 회원 목록이 아니다. 회원 정보, 시험 접수, 시험 상태, 결제 상태, 채점 결과, 피드백까지 한 화면에 섞여 있다. 따라서 실제 개발 관점에서는 `회원관리` 단일 화면이라기보다 `회원 중심 통합 조회 화면`으로 봐야 한다.

일반 사용자가 이해하기 쉽게 표현하면 다음과 같다.

- 관리자가 회원을 검색한다.
- 특정 조건에 맞는 회원만 필터링한다.
- 회원이 어떤 시험을 신청했는지 본다.
- 결제와 시험 상태를 같이 확인한다.
- 필요한 경우 엑셀로 내려받거나 파일로 올려 일괄 처리한다.
- 여러 명에게 메일이나 SMS를 보낸다.

### vibecoding 화면에서 강하게 보이는 요구사항

1. 대량 데이터 관리

   테이블 컬럼이 매우 많고 가로 스크롤이 보인다. 운영자가 많은 정보를 한 번에 비교해야 하는 화면이다.

2. 복합 필터

   체크박스가 많이 사용된다. 회원 유형, 가입 경로, 서비스 유형, 시험 상태, 요청 상태 같은 조건을 조합해서 검색하는 구조다.

3. 파일 기반 업무

   `파일올리기`, `파일다운로드`, `엑셀파일생성` 버튼이 보인다. 수동 입력보다 파일을 통한 대량 등록/수정/내보내기가 중요하다.

4. 관리자 업무 중심

   수험생이나 출제자가 직접 쓰는 화면이 아니라, 운영자가 전체 데이터를 관리하는 화면이다.

5. 상태 표시 중심

   `합격`, `불합격`, `접수완료`, `미응시`, `게시완료`, `취소` 같은 상태값을 화면에서 바로 확인하는 구조다.

## ai_docs IA 문서 분석

`ai_docs/ia분석파일.docx`는 화면 하나의 시안이 아니라 전체 서비스의 정보 구조를 정리한 문서다.

문서에서 확인되는 큰 역할은 다음 네 가지다.

| 역할 | 설명 |
|---|---|
| 수험생 | 시험 접수, 시험 응시, 결과 확인, 자격증 신청, 결제, 마이페이지, 문의 |
| 출제자 | 문제 출제, 출제 내역 확인, 반려 문제 수정, 최종 제출, 프로필 관리, 문의 |
| 채점자 | 배정 답안 확인, 채점 시작/이어하기, 채점 기준 확인, 채점 완료 내역, 재검토 |
| 관리자 | 회원, 시험, 출제, 채점, 결과, 자격증, 결제, 고객센터, 통계, 시스템 관리 |

### ai_docs가 정의하는 서비스 범위

`ai_docs`는 다음 내용을 상세히 다룬다.

- 홈 Dashboard
- 전체 메뉴 구조
- 수험생/출제자/채점자/관리자별 화면 분류
- 시험 접수, 응시, 결과 확인
- 문제 출제, 미리보기, 최종 제출, 관리자 검수
- 채점 기준표, 답안 채점, 최종 확정
- 회원 승인, 정지, 그룹 변경
- 시험 일정, 문제 검수, 문제은행, 결과 발표
- 결제, 환불, 정산
- 1:1 문의, 공지, FAQ
- 통계 관리
- 분류코드 관리
- 권한별 접근 제어
- 개인정보 마스킹
- 상태값 정의

### ai_docs의 핵심 방향

`ai_docs`의 핵심은 "역할별 업무 흐름을 설계하는 것"이다. 예를 들어 출제자는 문제를 만들고, 관리자는 검수하며, 채점자는 답안을 평가하고, 수험생은 시험을 접수하고 응시한다.

즉 `ai_docs`는 다음 질문에 답한다.

- 어떤 사용자가 있는가?
- 각 사용자는 어떤 메뉴를 쓰는가?
- 업무는 어떤 순서로 진행되는가?
- 어떤 상태값이 필요한가?
- 관리자는 어디까지 볼 수 있고 무엇을 승인해야 하는가?
- 개인정보와 권한은 어떻게 통제해야 하는가?

## vibecoding PDF와 ai_docs의 차이점

### 1. 문서의 목적 차이

| 비교 항목 | vibecoding PDF | ai_docs |
|---|---|---|
| 목적 | 실제 화면 UI 시안 확인 | 전체 서비스 IA와 업무 흐름 정의 |
| 성격 | 화면 중심 | 구조/정책/프로세스 중심 |
| 범위 | 관리자형 대량 관리 화면 중심 | 수험생, 출제자, 채점자, 관리자 전체 |
| 사용 시점 | 화면 구현, UI 컴포넌트 설계 시 유용 | 기획 확정, 메뉴 구조, DB/권한 설계 시 유용 |

### 2. 사용자 역할 범위 차이

`vibecoding PDF`는 관리자 화면이 중심이다. 화면 상단에 `관리자`가 표시되고, 회원관리/활동관리/결제관리/1:1문의/통계 같은 운영 메뉴가 보인다.

`ai_docs`는 관리자뿐 아니라 수험생, 출제자, 채점자까지 모두 다룬다. 그래서 전체 플랫폼을 만들려면 `ai_docs`가 더 넓은 기준이 된다.

### 3. 메뉴 구조 차이

`vibecoding PDF`의 대표 화면 탭은 다음처럼 보인다.

- 전체
- 회원관리
- 활동관리
- 결제관리
- 1:1문의 관리
- 통계

`ai_docs`의 관리자 메뉴는 더 넓다.

- 홈 Dashboard
- 회원관리
- 시험관리
- 출제관리
- 채점관리
- 결과관리
- 자격증관리
- 결제관리
- 고객센터
- 통계관리
- 시스템관리
- 분류코드

따라서 `vibecoding PDF`의 화면은 `ai_docs`의 관리자 IA 중 일부 메뉴를 한 화면에 통합한 형태로 보인다.

### 4. 데이터 표현 방식 차이

`vibecoding PDF`는 한 화면에 많은 컬럼을 넣는다. 회원 정보, 시험 정보, 결제 정보, 접수 정보, 채점 결과, 피드백이 한 테이블에 같이 표시된다.

`ai_docs`는 기능을 업무 단위로 나눈다. 예를 들어 회원은 회원관리, 시험은 시험관리, 결제는 결제관리, 결과는 결과관리로 나누는 식이다.

이 차이는 중요하다. 실제 구현 시 `vibecoding`처럼 모든 정보를 한 화면에 모으면 운영자는 빠르게 볼 수 있지만, 화면이 복잡해지고 권한/개인정보 관리가 어려워질 수 있다. `ai_docs`처럼 업무별로 나누면 구조는 명확하지만, 운영자가 여러 화면을 이동해야 할 수 있다.

### 5. 파일 업로드/엑셀 기능 차이

`vibecoding PDF`에서는 파일 관련 기능이 화면에 직접 보인다.

- 엑셀파일생성
- 파일올리기
- 파일다운로드
- 메일 보내기
- SMS 전송

`ai_docs`에도 엑셀/PDF 다운로드와 일괄 처리 필요성이 언급된다. 다만 `ai_docs`는 "어떤 화면에서 어떤 다운로드가 필요하다"는 정책에 가깝고, `vibecoding`은 실제 버튼과 배치가 보이는 UI 시안에 가깝다.

### 6. 보안/개인정보 관점 차이

`ai_docs`는 개인정보 보호를 명확히 언급한다.

- 비밀번호 컬럼 노출 금지
- 이름, 휴대폰, 이메일, 주소 마스킹
- 권한별 접근 제어
- 관리자 작업 로그 기록
- 삭제보다 정지/탈퇴 처리 권장

`vibecoding PDF` 화면은 이메일, 휴대폰, 주소, 생년월일, 접속 정보 등 민감 정보가 한 테이블에 많이 보인다. 따라서 `ai_docs` 기준을 적용하면 `vibecoding` 화면은 개인정보 마스킹과 권한 제한을 반드시 보완해야 한다.

### 7. 상태값 정의 차이

`vibecoding PDF`는 화면에 상태 배지나 텍스트가 직접 보인다.

- 정식회원
- 일반회원
- 회원 탈퇴
- 프리미엄
- 합격
- 불합격
- 접수완료
- 미응시
- 게시완료
- 취소

`ai_docs`는 상태값을 체계적으로 분류한다.

- 회원 상태: 승인대기, 승인완료, 반려, 정지, 탈퇴, 삭제대기
- 시험 상태: 일정등록, 접수예정, 접수중, 접수마감, 시험진행중, 채점중, 결과확정, 결과공개
- 문제 상태: 출제중, 제출완료, 검수대기, 수정요청, 반려, 확정, 배포완료
- 결제 상태: 결제대기, 결제완료, 결제실패, 취소완료, 환불대기, 환불완료, 정산완료

실제 개발에서는 `ai_docs`의 상태값을 기준으로 삼고, `vibecoding` 화면의 표시 문구를 그 상태값에 매핑하는 방식이 적절하다.

## 기능별 상세 비교

| 기능 영역 | vibecoding PDF | ai_docs | 판단 |
|---|---|---|---|
| 회원관리 | 실제 고밀도 테이블과 필터가 있음 | 회원 승인/반려, 정지/복구, 그룹 변경, 상세보기, 일괄 처리 정의 | `ai_docs` 정책을 기준으로 `vibecoding` UI를 보완 |
| 활동관리 | 탭으로 존재 | 출제/채점/응시 활동 흐름을 역할별로 설명 | `vibecoding`은 운영 조회 화면, `ai_docs`는 업무 흐름 설명 |
| 결제관리 | 탭과 상태 컬럼이 보임 | 매출, 환불, 정산, PG 내역, 보고서 정의 | 결제 상세 정책은 `ai_docs`가 더 명확 |
| 1:1문의 | 탭으로 존재 | 문의 등록, 답변, 템플릿, 처리 이력 정의 | `vibecoding`은 목록 통합, `ai_docs`는 처리 프로세스까지 포함 |
| 통계 | 탭으로 존재 | 회원/시험/결제/합격자/자격증/교재 통계 정의 | 통계 항목은 `ai_docs` 기준 필요 |
| 시험관리 | 테이블 컬럼에 시험명, 시험상태, 시험결과 등이 보임 | 일정, 문제, 응시, 결과, 자격증까지 전체 흐름 정의 | `vibecoding`은 시험 결과 조회 일부만 반영 |
| 출제관리 | 대표 화면에서는 직접 메뉴로 분리되지 않음 | 문제 출제, 미리보기, 최종 제출, 관리자 검수 상세 정의 | `vibecoding`만으로는 출제자 화면 구현 불가 |
| 채점관리 | 결과/채점 피드백 컬럼 일부 보임 | 채점 기준표, 답안 유형별 채점, 최종 확정 정의 | 채점 화면은 `ai_docs`를 따라 별도 설계 필요 |
| 권한관리 | 화면상 명확한 권한 정책은 보이지 않음 | 최고관리자, 회원관리자, 시험관리자, 결제관리자 등 권한 정의 | 운영 안정성을 위해 `ai_docs` 기준 필수 |
| 파일 처리 | 화면에 버튼이 명확히 보임 | 일괄 처리/다운로드 필요성 언급 | UI 배치는 `vibecoding`, 정책은 `ai_docs` 결합 |

## 개발 관점에서의 해석

실제 구현 기준은 다음처럼 잡는 것이 좋다.

1. 전체 IA와 권한은 `ai_docs`를 기준으로 한다.

   서비스 전체 역할과 메뉴 구조가 `ai_docs`에 더 잘 정의되어 있다. 수험생, 출제자, 채점자, 관리자 흐름을 모두 구현하려면 `ai_docs`가 기준 문서가 되어야 한다.

2. 관리자 고밀도 테이블 UI는 `vibecoding PDF`를 기준으로 한다.

   필터, 체크박스, 테이블, 파일 업로드/다운로드, 엑셀 생성 버튼 배치는 `vibecoding` 시안이 더 구체적이다.

3. 개인정보와 권한은 `ai_docs` 기준으로 보완한다.

   `vibecoding` 화면에는 민감 정보가 많이 노출된다. 실제 운영 화면에서는 권한별 컬럼 노출, 마스킹, 다운로드 제한, 작업 로그가 필요하다.

4. 상태값은 `ai_docs`의 표준 상태값으로 정리한다.

   화면에 보이는 `미응시`, `접수완료`, `합격`, `불합격` 같은 문구를 시스템 상태값과 매핑해야 한다.

5. `vibecoding`은 전체 서비스가 아니라 관리자 통합 조회/파일 처리 모듈로 분리한다.

   메뉴상으로는 `관리자 > 회원관리` 또는 `관리자 > 통합관리` 하위 기능으로 두는 것이 자연스럽다.

## 보완해야 할 점

### vibecoding PDF 기준 보완점

- 화면에 정보가 너무 많아 일반 관리자가 어떤 작업을 먼저 해야 하는지 어렵다.
- 회원 정보와 시험 정보, 결제 정보가 한 테이블에 섞여 있어 권한 관리가 복잡하다.
- 개인정보 마스킹 기준이 화면에서 명확히 보이지 않는다.
- 파일 업로드 후 어떤 검증/오류 처리/반영 절차가 있는지 확인이 필요하다.
- `구분` PDF들이 어떤 순서의 화면인지, 같은 화면의 상태별 시안인지 추가 확인이 필요하다.

### ai_docs 기준 보완점

- 문서 내용은 매우 넓지만 실제 화면 배치까지는 부족하다.
- 각 메뉴의 실제 URL, 화면 ID, 컴포넌트 단위가 정리되어 있지 않다.
- `vibecoding`에 보이는 파일 업로드형 대량 처리 화면이 IA 어디에 들어가는지 명확히 연결해야 한다.
- 관리자 통합 조회 화면처럼 여러 업무 데이터를 합쳐 보는 화면의 정책이 더 필요하다.

## 최종 판단

`vibecoding PDF`는 관리자용 실무 화면의 구체적인 UI 참고 자료다. 특히 대량 필터, 고밀도 테이블, 파일 업로드/다운로드, 엑셀 생성 같은 운영 기능을 구현할 때 유용하다.

`ai_docs`는 서비스 전체 설계 기준이다. 사용자 역할, 메뉴 구조, 업무 순서, 상태값, 권한, 보안 정책은 `ai_docs`를 따라야 한다.

따라서 두 자료는 서로 대체 관계가 아니라 보완 관계다. 구현 기준은 `ai_docs`로 잡고, 실제 관리자 화면의 UI와 파일 처리 방식은 `vibecoding PDF`를 반영하는 방식이 가장 적절하다.

## 추천 적용 순서

1. `ai_docs` 기준으로 전체 메뉴와 권한 구조를 먼저 확정한다.
2. 관리자 메뉴 안에 `vibecoding`의 통합 조회/파일 업로드 화면이 들어갈 위치를 정한다.
3. `vibecoding` 화면의 컬럼을 개인정보, 시험, 결제, 결과, 피드백 영역으로 재분류한다.
4. 개인정보 컬럼은 권한별 마스킹 규칙을 적용한다.
5. 파일 업로드/다운로드는 작업 로그와 검증 결과 화면을 함께 설계한다.
6. 화면에 보이는 상태 문구를 `ai_docs`의 표준 상태값과 매핑한다.
7. 마지막으로 `구분-1` ~ `구분-14` PDF의 순서를 확인해 세부 플로우로 정리한다.

## UX/UI 관점 보완

`ai_docs`는 구조를 잘 설명하지만 실제 화면의 복잡도를 보여주지는 않는다. 반대로 `vibecoding`은 실제 관리자 테이블의 밀도를 보여주지만, 사용자가 처음 봤을 때 어떻게 이해해야 하는지에 대한 안내가 부족하다.

| 항목 | ai_docs | vibecoding | UX/UI 보완 방향 |
|---|---|---|---|
| 메뉴 이해도 | 역할별 메뉴가 명확함 | 한 화면에 여러 업무 탭이 모임 | 상위 IA는 ai_docs를 따르고, vibecoding은 관리자 고급 조회 화면으로 둔다. |
| 화면 복잡도 | 비교적 설명 중심 | 컬럼과 필터가 매우 많음 | 기본 보기와 고급 보기를 분리한다. |
| 데이터 처리 | 기능 요구 중심 | 파일 업로드/다운로드가 강함 | 업로드 검증, 미리보기, 반영 이력 UX를 추가한다. |
| 개인정보 | 보안 필요성을 언급 | 개인정보가 테이블에 직접 노출될 가능성 | 마스킹, 권한별 컬럼, 다운로드 로그가 필요하다. |
| 대량 작업 | 구체성이 약함 | 선택삭제, 메일, SMS, 엑셀 처리 가능 | 대량 작업 전 대상 수와 변경 내용을 확인시킨다. |

### 화면 적용 원칙

1. `vibecoding` 화면을 관리자 기본 화면으로 쓰지 않는다.
2. 기본 화면은 요약 카드와 주요 미처리 항목 중심으로 단순하게 둔다.
3. `vibecoding`식 테이블은 `고급 검색`, `통합 조회`, `엑셀 처리` 같은 별도 메뉴로 제공한다.
4. 필터 조건은 저장 가능한 프리셋으로 만든다. 예: `미응시자`, `결제 미완료`, `합격 후 자격증 미신청`.
5. 파일 업로드는 검증 결과를 먼저 보여주고, 관리자가 확인한 뒤 반영한다.
