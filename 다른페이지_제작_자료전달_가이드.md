# 다른 페이지 제작 시 자료 전달 가이드

이 문서는 새 HTML 페이지나 새 화면을 만들 때 Markdown 문서, HTML, CSS, JS, 이미지 자료를 프롬프트에 어떻게 전달하면 좋은지 정리한 가이드다.

핵심은 "많이 붙여넣기"가 아니라 "자료의 역할과 우선순위를 분명하게 전달하기"다.

## 1. 가장 좋은 전달 순서

1. 만들 화면의 목적
2. 최종 산출물 형식
3. 반드시 따를 기준 Markdown
4. 참고할 HTML/CSS/JS 파일
5. 화면에 들어갈 실제 콘텐츠
6. 기존 파일에서 유지해야 할 기능
7. 금지할 디자인/구현 방식
8. 검수 기준

예시는 다음과 같다.

```text
목표:
- 관리자용 결제관리 화면을 새로 만든다.
- 기존 디자인시스템의 BO 테이블 밀도와 Drawer 방식을 따른다.

최종 산출물:
- 새 파일: 디자인시스템/table_design/결제관리_v2.html
- 단일 HTML로 작성하되, 기존 table-design.css와 table-design.js를 재사용한다.

반드시 읽을 기준 문서:
- 디자인시스템/docs/재사용_디자인_프롬프트/관리자/02_BO_테이블레이아웃_기준.md
- 디자인시스템/docs/재사용_디자인_프롬프트/관리자/components/검색필터_기준.md
- 디자인시스템/docs/재사용_디자인_프롬프트/관리자/components/테이블선택_Drawer_상세패널_기준.md

참고 HTML:
- 디자인시스템/table_design/결제관리.html
- 디자인시스템/table_design/관리자패널_전체회원.html

유지할 것:
- 고밀도 테이블
- 행 선택 후 우측 상세 패널
- 상태 chip 표기
- 상단 검색필터

금지:
- 마케팅 랜딩페이지처럼 큰 카드와 넓은 여백 사용 금지
- 새로운 컬러 팔레트 임의 생성 금지
- 상태값을 일반 텍스트만으로 표시하지 말 것

검수:
- 데스크톱에서 한 화면에 표와 상세 패널이 함께 보여야 한다.
- 모바일에서는 표가 가로 스크롤되거나 상세가 아래로 자연스럽게 내려가야 한다.
```

## 2. Markdown 기준 문서를 전달하는 방법

Markdown 문서는 "전체를 다 읽어라"보다 "어떤 역할로 쓰는지"를 함께 알려줘야 한다.

좋은 방식:

```text
아래 문서는 디자인 기준으로 사용해줘.
- 02_BO_테이블레이아웃_기준.md: 테이블 밀도, 컬럼, 필터, 행 액션 기준
- 검색필터_기준.md: 상단 필터 UI 기준
- 테이블선택_Drawer_상세패널_기준.md: 행 클릭 후 상세 패널 기준

이 문서들의 문구를 화면에 노출하지 말고, 레이아웃/컴포넌트 규칙으로만 반영해줘.
```

나쁜 방식:

```text
이 md들 참고해서 예쁘게 만들어줘.
```

이렇게 쓰면 어떤 문서를 우선해야 하는지, 문서를 화면 콘텐츠로 써야 하는지, 디자인 규칙으로 써야 하는지 모호해진다.

## 3. HTML/CSS/JS 파일을 프롬프트로 전달하는 방법

HTML 파일은 크게 3가지 역할로 나눠 전달하는 것이 좋다.

| 전달 역할 | 설명 | 예시 |
|---|---|---|
| 기준 화면 | 새 화면이 따라야 할 구조와 밀도 | `관리자패널_전체회원.html` |
| 참고 화면 | 일부 패턴만 참고 | `결제관리.html`의 상태칩 |
| 콘텐츠 원천 | 문구, 표 데이터, 메뉴명만 가져올 자료 | 기존 랜딩페이지 HTML |

전달 예시:

```text
참고 파일 역할:
- 디자인시스템/table_design/관리자패널_전체회원.html: 전체 레이아웃 기준
- 디자인시스템/table_design/결제관리.html: 결제 상태 chip과 필터 항목 참고
- 디자인시스템/table_design/table-design.css: 스타일 재사용
- 디자인시스템/table_design/table-design.js: 행 선택/상세 패널 동작 재사용

주의:
- 참고 HTML의 더미 데이터는 그대로 복사하지 말고, 새 화면 목적에 맞게 교체한다.
- CSS 클래스는 기존 것을 우선 사용한다.
- 새 CSS는 기존 구조로 해결이 안 되는 경우에만 추가한다.
```

## 4. 큰 파일을 전달할 때의 권장 방식

루트 `index.html`처럼 파일이 큰 경우에는 전체 파일을 프롬프트에 붙여넣지 않는 편이 좋다.

대신 다음 중 하나로 전달한다.

### 4.1 로컬 파일 경로로 전달

같은 작업 폴더 안에서 작업할 때 가장 좋다.

```text
파일을 직접 열어서 확인해줘.
- 대상: index.html
- 수정 대상: DATA.htmlDocs, DATA.mdDocs의 section 값
- 유지 대상: CSS, render 함수, Markdown content
```

### 4.2 필요한 코드 조각만 전달

외부 도구에 붙여넣어야 할 때는 필요한 부분만 준다.

```text
아래 부분만 수정하면 된다.
- const DATA = { ... } 안의 htmlDocs, mdDocs 배열
- 각 항목의 section 값

수정하지 말 것:
- content 필드의 Markdown 원문
- renderMarkdown 함수
- 이벤트 리스너
```

### 4.3 파일 트리와 기준 문서 목록을 먼저 전달

새 페이지를 만들 때는 전체 코드보다 파일 구조가 더 중요할 때가 많다.

```text
관련 파일 구조:
- 디자인시스템/docs/재사용_디자인_프롬프트/00_문서목록.md
- 디자인시스템/docs/재사용_디자인_프롬프트/관리자/02_BO_테이블레이아웃_기준.md
- 디자인시스템/table_design/table-design.css
- 디자인시스템/table_design/table-design.js
- 디자인시스템/table_design/관리자패널_전체회원.html

먼저 문서목록을 읽고, 필요한 기준 문서만 골라 적용해줘.
```

## 5. 그룹 다운로드 자료를 전달하는 방법

`index.html`에 그룹별 다운로드 기능이 붙으면, 새 페이지를 만들 때는 개별 파일을 하나씩 나열하지 않고 그룹 zip을 기준 자료로 전달할 수 있다.

예시:

```text
참고 자료는 index.html의 "디자인시스템/재사용 UI" 그룹 다운로드 zip을 기준으로 사용해줘.

zip 안에서 우선 읽을 문서:
- 00_문서목록.md
- 관리자/02_BO_테이블레이아웃_기준.md
- 관리자/components/검색필터_기준.md
- 관리자/components/테이블선택_Drawer_상세패널_기준.md

HTML 기준:
- table_design/관리자패널_전체회원.html
- table_design/결제관리.html

주의:
- zip 전체를 그대로 복사하지 말고, 새 화면에 필요한 구조와 컴포넌트 기준만 반영해줘.
- CSS/JS는 기존 기준 파일을 우선 재사용해줘.
```

그룹 zip을 전달할 때도 "zip 안의 어떤 파일이 기준인지"를 꼭 지정해야 한다. 그렇지 않으면 분석 자료, 원본 자료, 실제 기준 파일이 같은 우선순위로 해석될 수 있다.

## 6. 압축파일을 해제해서 전달하는 방법

zip 파일을 이미 압축 해제해서 폴더 형태로 전달했다면, zip 파일명보다 `압축 해제된 기준 폴더`와 `먼저 읽을 문서`를 명확히 지정하는 것이 좋다.

좋은 방식:

```text
아래 폴더는 index.html의 "디자인시스템/재사용 UI" 그룹 zip을 압축 해제한 기준 자료야.

기준 폴더:
- downloads/unzipped/디자인시스템_재사용_UI/

먼저 읽을 문서:
1. 디자인시스템/docs/재사용_디자인_프롬프트/00_문서목록.md
2. 디자인시스템/docs/재사용_디자인_프롬프트/관리자/02_BO_테이블레이아웃_기준.md
3. 디자인시스템/docs/재사용_디자인_프롬프트/관리자/components/검색필터_기준.md
4. 디자인시스템/docs/재사용_디자인_프롬프트/관리자/components/테이블선택_Drawer_상세패널_기준.md

참고 HTML:
1. 디자인시스템/table_design/관리자패널_전체회원.html
2. 디자인시스템/table_design/결제관리.html

사용 방식:
- 압축 해제 폴더 전체를 새 프로젝트에 그대로 복사하지 말고, 필요한 기준과 컴포넌트 구조만 참고해줘.
- CSS/JS는 기준 파일의 클래스와 동작을 우선 재사용해줘.
- old, origin, resource 성격의 파일은 명시적으로 요청하지 않는 한 우선순위를 낮춰줘.
```

나쁜 방식:

```text
압축 풀어놨으니까 이 폴더 보고 만들어줘.
```

이렇게 전달하면 어떤 파일이 기준이고 어떤 파일이 참고용인지 구분하기 어렵다.

### 6.1 압축 해제 폴더 전달 시 권장 구조

압축 해제 폴더는 원본 zip 이름과 같은 이름으로 두는 것이 좋다.

```text
downloads/unzipped/
├─ 디자인시스템_재사용_UI/
├─ 관리자_BO_업무화면/
├─ 사용자_랜딩_신청_화면/
├─ 내부_업무도구_자동화/
├─ 분석/
├─ 기획_전략_자료/
├─ 교육_시험_콘텐츠_자료/
├─ 개발_운영_매뉴얼/
└─ 보관_원본_자료/
```

이렇게 두면 프롬프트에서 다음처럼 짧게 지시할 수 있다.

```text
기준 자료는 downloads/unzipped/관리자_BO_업무화면/ 폴더를 사용해줘.
먼저 00_문서목록 또는 관리자 기준 문서를 읽고, 그 다음 HTML 샘플을 참고해줘.
```

### 6.2 zip 전달과 압축 해제 폴더 전달의 차이

| 방식 | 장점 | 주의점 |
|---|---|---|
| zip 파일 전달 | 하나의 파일로 주고받기 쉽다 | 작업자가 압축을 풀어야 내부 파일을 볼 수 있다 |
| 압축 해제 폴더 전달 | 내부 파일을 바로 읽고 비교하기 쉽다 | 폴더 안의 우선순위를 반드시 지정해야 한다 |

압축 해제 폴더를 전달할 때는 다음 3가지를 꼭 함께 적는다.

- 기준 폴더 경로
- 먼저 읽을 Markdown 문서
- 참고할 HTML/CSS/JS 파일

### 6.3 압축 해제 폴더 기반 요청 템플릿

```text
[압축 해제 기준 자료]
- 원본 zip:
- 압축 해제 폴더:

[먼저 읽을 문서]
1.
2.
3.

[참고할 화면/코드]
1.
2.
3.

[사용 규칙]
- 기준 문서는 레이아웃/컴포넌트 규칙으로 반영한다.
- HTML 샘플의 더미 데이터는 그대로 복사하지 않는다.
- CSS 클래스와 JS 동작은 기존 기준을 우선 사용한다.
- old/origin/resource 폴더는 보관 자료로 보고, 명시된 경우에만 참고한다.

[만들 산출물]
- 파일 경로:
- 화면 목적:
- 유지할 기능:
- 금지할 방식:
```

## 8. 관리자 화면 요청 예시

```text
디자인시스템 기준을 따라 관리자용 "시험 접수 관리" 화면을 만들어줘.

최종 파일:
- 디자인시스템/table_design/시험접수관리.html

반드시 적용할 기준:
- 디자인시스템/docs/재사용_디자인_프롬프트/관리자/02_BO_테이블레이아웃_기준.md
- 디자인시스템/docs/재사용_디자인_프롬프트/관리자/04_BO_관리자용_레이아웃_기준.md
- 디자인시스템/docs/재사용_디자인_프롬프트/관리자/components/테이블선택_Drawer_상세패널_기준.md

참고 화면:
- 디자인시스템/table_design/관리자패널_전체회원.html
- 디자인시스템/table_design/결제관리.html

화면 구성:
```
