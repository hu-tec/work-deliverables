## 1. 한 줄 진단

지식·규정, 일상 업무, 인사/채용, 운영 관리, 사이트 관리까지 **성격이 다른 업무 도메인이 한 사이드바에 같은 층위로 약 49개 평면 나열**되어 있다. 같은 데이터의 여러 화면(특히 규정 6~8종)이 별도 메뉴로 흩어져 있고, 이전/대체 메뉴까지 함께 노출되어 "내 업무를 어디서 시작하는지"가 보이지 않는다.

---

## 2. 현재 메뉴 구조 (AS-IS)

분석 시점 기준 관찰된 그룹과 메뉴 수다. 총 **10개 그룹 / 약 49개 경로**.

| # | 현재 그룹 | 메뉴 수 | 대표 메뉴 |
| --- | --- | ---: | --- |
| 1 | 업무 | 9 | 사내업무지침, 업무 총괄, 업무 자료, 업무일지, 서약서, 레슨플랜, 매뉴얼 리스트 |
| 2 | 채용/인사 | 3 | 채용관리, 출퇴근 관리 A, 출퇴근 관리 B |
| 3 | 학생관리 | 1 | 학생관리 |
| 4 | 관리/운영 | 11 | 거래처 아웃콜, 사진모음, 미팅, 강의시간표, 미수금관리, 관리자통합, 거래처관리, 프로젝트관리 |
| 5 | 규정 | 8 | 규정 통합, 규정관리, 규정편집, 규정 레이아웃, 규정매뉴얼, 평가기준 |
| 6 | 서식/확인서 | 1 | 서식/확인서 |
| 7 | 출강관리 | 2 | 출강강사관리, 거래처관리(출장) |
| 8 | 기획/업무기획 | 5 | 업무 분류, 강사커리, 마케팅, 프롬사용법, AI 강의안 |
| 9 | 커뮤니티/홈페이지 | 2 | 커뮤니티, 홈페이지 |
| 10 | 대체·이전 메뉴 | 7 | 업무일지/사내지침/면접 관련 이전 화면의 대체 링크 |

### 문제점 5가지

1. **층위 혼재** — 일반 업무(업무일지), 민감 인사 데이터(출퇴근·채용), 고권한 시스템 설정(관리자통합)이 같은 깊이에 나란히 있어 권한 경계가 시각적으로 드러나지 않는다.
2. **중복 메뉴** — `규정 통합 / 규정관리 / 규정편집 / 규정 레이아웃 / 규정매뉴얼 / 평가기준`은 사실상 같은 규정 데이터의 다른 보기/모드다. 6~8개 메뉴로 쪼개져 사이드바 길이를 키운다.
3. **이전 메뉴 노출** — `대체·이전 메뉴` 7개가 현역 메뉴와 섞여 있어 사용자가 옛 화면을 잘못 사용할 위험이 있다.
4. **그룹 과다** — 그룹이 10개라 접힘(accordion) 없이 모두 펼쳐지면 한 화면을 넘는 스크롤이 발생한다.
5. **진입점 부재** — 로그인 직후 "오늘 내가 할 일"로 데려가는 개인 허브/대시보드가 사이드바 최상단에 없다.

## 3. 단순화 원칙

| 원칙 | 설명 |
| --- | --- |
| ① 5±2 규칙 | 평상시 보이는 최상위 항목을 7개 이하로 — 대그룹 5개 + 즐겨찾기/시스템/보관함 |
| ② 데이터 1개 = 메뉴 1개 | 같은 데이터의 여러 보기는 메뉴를 늘리지 말고 화면 안의 탭/뷰 전환으로 처리 (규정 6종 → 규정 1개) |
| ③ 권한으로 분리 | 고권한(시스템 관리)·민감 인사 영역은 별도 섹션 + 역할 기반 표시로 분리 |
| ④ 역할 기반 기본값 | 로그인한 역할에 맞는 그룹만 펼치고 나머지는 접어 둠 |
| ⑤ 즐겨찾기 + 검색 | 자주 쓰는 5~7개를 상단에 고정, 상단 검색으로 49개를 한 번에 점프 |
| ⑥ 이전 메뉴는 보관함으로 | 현역에서 분리해 `보관함(Legacy)`에 숨김 (기본 접힘) |

## 4. 개선 구조 (TO-BE)

**최상위 5개 대그룹 + 즐겨찾기 + 시스템 관리 + 보관함**. 각 대그룹은 접이식(accordion) 섹션이고, 평상시엔 헤더만 보인다.

```text
🔍 [검색…]                         ← 상단 검색 (49개 전체 점프)
⭐ 즐겨찾기                          ← 사용자가 고정한 5~7개
─────────────────────────────
1. 🏠 홈 · 내 업무
     • 대시보드(개인 허브)
     • 업무일지
     • 업무 총괄
     • 내 출퇴근
2. 📚 지식 · 규정
     • 사내업무지침
     • 규정            ← 통합/관리/편집/레이아웃/매뉴얼/평가기준을 내부 탭으로 통합
     • 매뉴얼 리스트
     • 서식 · 확인서
     • 업무 자료 / 서약서 / 레슨플랜
3. 👥 인사 · 채용
     • 채용관리        ← 지원서/면접/대시보드/평가를 내부 탭으로 통합
     • 출퇴근 관리      ← A/B 통합
     • 학생관리
     • 출강 · 강사관리
4. 🏢 운영 · 거래처
     • 미팅 / 강의시간표
     • 거래처관리       ← 거래처관리 + 아웃콜 + 출장 거래처 통합
     • 미수금관리
     • 프로젝트관리
     • 사진모음
5. 🧭 기획 · 콘텐츠
     • 업무 분류 / 강사커리
     • 마케팅
     • 프롬프트 사용법 / AI 강의안
     • 커뮤니티 / 홈페이지
─────────────────────────────
⚙️ 시스템 관리   (관리자 전용·권한 분리)
     • 관리자통합
🗄️ 보관함(Legacy)  (기본 접힘·이전 메뉴 7종)
```

### 통합 매핑 표 (49개 → 정리)

| TO-BE 대그룹 | 흡수한 AS-IS 메뉴 | 통합 방식 |
| --- | --- | --- |
| 1. 홈·내 업무 | 업무일지, 업무 총괄, 출퇴근(개인 보기) + 신규 대시보드 | 개인 허브 신설, 출퇴근은 "내 기록"만 노출 |
| 2. 지식·규정 | 사내업무지침, 규정 통합/관리/편집/레이아웃/매뉴얼/평가기준, 매뉴얼 리스트, 서식/확인서, 업무 자료, 서약서, 레슨플랜 | **규정 6~8종 → 화면 내 탭 1개로 통합** |
| 3. 인사·채용 | 채용관리(+지원서/면접/대시보드/강사평가), 출퇴근 A/B, 학생관리, 출강강사관리 | 채용 하위 화면을 탭으로, 출퇴근 A/B 단일화 |
| 4. 운영·거래처 | 거래처관리, 거래처 아웃콜, 거래처관리(출장), 미수금관리, 미팅, 강의시간표, 프로젝트관리, 사진모음 | 거래처 3종 → 1개(탭) |
| 5. 기획·콘텐츠 | 업무 분류, 강사커리, 마케팅, 프롬사용법, AI 강의안, 커뮤니티, 홈페이지 | 그대로 묶음 |
| 시스템 관리 | 관리자통합 | 권한 분리·하단 고정 |
| 보관함 | 대체·이전 메뉴 7종 | 기본 숨김 |

### 효과 (정량)

| 지표 | AS-IS | TO-BE |
| --- | ---: | ---: |
| 최상위 그룹 수 | 10 | 5 (+부가 3) |
| 평상시 보이는 항목 | ≈49 (전부 펼침) | ≈8 (헤더 + 즐겨찾기) |
| 규정 관련 메뉴 | 6~8 | 1 (내부 탭) |
| 거래처 관련 메뉴 | 3 | 1 (내부 탭) |
| 이전/대체 메뉴 노출 | 7 (현역과 혼재) | 0 (보관함 격리) |

## 5. 구현 가이드 (shadcn/ui 사이드바 기준)

현재 앱은 shadcn/ui 사이드바를 쓰므로 다음 컴포넌트로 그대로 매핑된다.

1. **대그룹 = `SidebarGroup` + 접이식 `Collapsible`** — `SidebarGroupLabel`을 트리거로, 하위는 `SidebarMenuSub`.
2. **즐겨찾기 = 별도 최상단 `SidebarGroup`** — 사용자별 `localStorage`/서버 저장, 각 메뉴 우측 `☆` 토글.
3. **검색 = `SidebarHeader` 내 입력 + `Command` 팔레트** — 라우트 메타데이터(라벨·그룹·권한)로 필터.
4. **권한 분리** — 라우트 메타에 `role`/`scope`를 부여하고, `시스템 관리`·민감 인사 메뉴는 권한 없으면 렌더 자체를 막는다(화면 숨김 + 서버 가드 병행).
5. **데이터 통합(규정/채용/거래처)** — 라우트를 합치고 화면 내 `Tabs`로 보기 전환. 기존 URL은 새 탭 쿼리로 301/리다이렉트해 북마크 보존.
6. **보관함** — 하단 접이식 `SidebarGroup`(기본 닫힘), 라벨에 `Legacy` 배지.

### 라우트 메타데이터 예시

```ts
// 메뉴를 데이터로 정의하면 그룹화·검색·권한·즐겨찾기를 한 곳에서 관리
type NavItem = {
  id: string;
  label: string;
  path: string;
  group: '홈' | '지식규정' | '인사채용' | '운영거래처' | '기획콘텐츠' | '시스템' | 'legacy';
  roles?: string[];     // 없으면 전체 공개
  legacy?: boolean;     // true면 보관함으로
  tabOf?: string;       // 다른 화면의 내부 탭이면 부모 id
};
```

## 6. 적용 순서 (권장)

| 단계 | 작업 | 리스크 |
| --- | --- | --- |
| 1 | 라우트를 `NavItem[]` 데이터로 추출 (현행 그대로) | 낮음 — 표시 로직만 분리 |
| 2 | 5개 대그룹 + 접이식으로 사이드바 렌더 교체 | 낮음 — URL 불변 |
| 3 | 이전/대체 메뉴 7종 → 보관함으로 이동 | 중간 — 사용자 공지 필요 |
| 4 | 규정·채용·거래처를 화면 내 탭으로 통합 | 중간 — 기존 URL 리다이렉트 |
| 5 | 즐겨찾기 + 검색 팔레트 추가 | 낮음 |
| 6 | 역할 기반 표시 + 서버 권한 가드 | 높음 — 인가 로직 검증 필수 |

> 디자인(직각·플랫) 개선은 별도 문서 [02-디자인-개선안(border-radius).md](./02-디자인-개선안(border-radius).md) 참고. 메뉴 구조와 독립적으로 적용 가능하다.
